# Product Requirements Document (V1.5) - Real-World Production Capacity Modeler

## 1. Product Goal
The goal of V1.5 is to make the capacity modeling tool usable for **80% of real production lines** by supporting complex network topologies (DAGs with merges and splits) while maintaining simplicity and clarity. V1.5 enables users to model assembly lines, material splits, scrap paths, and multi-step production flows that reflect actual manufacturing operations.

## 2. Target Users
- Manufacturing Engineers
- Operations Managers
- Industrial Engineers
- Plant Managers

## 3. Core Workflow (Enhanced)
1. **Model:** User connects nodes in complex arrangements, including merges (assembly) and splits (branching/scrap/distribution).
2. **Configure:** User defines node parameters (throughput, available time, yield) and split/merge ratios.
3. **Input:** User enters global demand tied to the Sink output material.
4. **Analyze:** System propagates demand upstream through complex networks and calculates utilization, bottlenecks, and required throughput per node.
5. **Scenario Test:** User duplicates and modifies scenarios to test "what-if" questions (add a machine, reduce scrap, change demand).

## 4. V1.5 Scope (MVP for 80% of Production Lines)

### Must Have:
- **Non-Linear Graph Support (DAGs):** Support for merges (many-to-one assembly) and splits (one-to-many distribution).
- **Bill of Materials (BOM) on Merges:** Define how multiple input materials combine (e.g., 4 units of Part A + 1 unit of Part B = 1 unit of Assembly).
- **Split Ratios:** Define how output volume is distributed when a node has multiple outputs (e.g., 95% to good output, 5% to scrap).
- **Complex Demand Propagation:** Use topological sort to correctly propagate demand upstream through splits and merges.
- **User Authentication:** Secure login so users own their data; anonymous access is removed.
- **Import/Export:** Users can import existing capacity models from Excel/CSV and export results as spreadsheets or PDF.
- **Improved Validation & Error Handling:** Clear feedback when graphs are invalid (cycles, improper merges) and protection against data loss.

### Explicitly Deferred (Post-MVP):
- **Operational Calendars:** Users manually enter "Available Time" per node (keeps MVP simple).
- **Max Capability Mode:** Defer advanced "find the ceiling" analysis until MVP is proven with real users.
- **Side-by-side Scenario Comparison:** Focus on save/load/duplicate workflow first.

## 5. Main Screens (Updates)
- **Login Screen:** Authentication required before accessing models.
- **Dashboard:** List of user's saved models with basic filters (date, name). Add ability to upload Excel model files.
- **Editor:** Canvas with support for merge and split node configuration. Right sidebar now includes merge BOM inputs and split ratio inputs.
- **Results Panel:** Shows required throughput, utilization, and bottleneck for each node. Updated to handle complex networks correctly.
- **Export Modal:** Download results as spreadsheet or PDF report.

## 6. Graph Constraints (Updated)
Allowed in V1.5:
- **Merging (Assembly):** Multiple inputs into one Process or Sink node.
- **Branching (Split):** Multiple outputs from one Source or Process node.
- **DAGs:** Any directed acyclic structure.

Still Not Allowed:
- **Cycles/Loops:** No circular flows or rework loops that create infinite recursions.

## 7. Node Configuration (Updates)

### All Nodes (Existing from V1):
- Name, Throughput Rate, Available Time, Yield, Number of Resources

### Merge Nodes (New):
- **Bill of Materials (BOM):** For each input edge, define the quantity required per unit of output.
  - Example: Node requires "4 units from Source A" + "1 unit from Source B" to produce "1 unit of Assembly".
  - Stored on the merge node itself, with one ratio entry per incoming edge.
  - The system uses these ratios during upstream demand propagation.

### Split Nodes (New):
- **Split Ratios:** For each output edge, define the percentage of volume distributed to that path.
  - Example: "95% to Path A (good output)" and "5% to Path B (scrap)".
  - Split ratios must sum to 100%.
  - The system distributes demand accordingly during propagation.

## 8. Calculation Engine (V1.5 Math)

### Complex Demand Propagation:
1. **Topological Sort:** Order nodes from Sink to Source to ensure correct propagation direction.
2. **Upstream Propagation:** Starting at the Sink with global demand, work backward through the graph.
3. **Merge Handling:** When a node has multiple inputs, use the BOM ratios to calculate required input from each upstream source.
   - Example: If downstream requires 100 units and BOM is "4A + 1B", then require 400 units from Source A and 100 units from Source B.
4. **Split Handling:** When a node has multiple outputs, distribute the required input based on split ratios.
   - Example: If split is "95% / 5%", then a node with 100 units output must accept 105 upstream units to account for the 5% loss.
5. **Yield Application:** At each node, apply yield losses to calculate required input from the previous step.

### Bottleneck & Utilization:
- For each node, calculate Utilization = Required Throughput / Effective Capacity.
- Identify the bottleneck as the node with the highest utilization.
- System Throughput = the bottleneck-limited throughput expressed in Sink output units.

## 9. What is Out of Scope for V1.5
- **Operational Calendars:** Deferred to post-MVP. Users enter "Available Time" manually (simpler MVP).
- **Max Capability Mode:** Deferred to post-MVP. Users provide global demand; system propagates it.
- **Shift Templates:** Deferred to post-MVP. No automatic shift calculation.
- **Stochastic Simulation, Queues, Scheduling:** Still out of scope (V1.5 remains deterministic).
- **Cycles/Rework Loops:** Not allowed; system must remain a DAG.
- **Multi-Product Simultaneous Optimization:** Single product flow per model for V1.5.

## 10. Success Criteria for V1.5 (80% Production Coverage)

### Core Modeling:
- ✅ User can model a **3-part assembly process** (3 Sources → 3 Process nodes → 1 Merge → 1 Sink).
- ✅ User can model a **production split** (1 Source → 1 Process → 2 outputs: 95% good, 5% scrap).
- ✅ User can model a **complex network**: Sources → Processes → Assembly (merge) → Finish → Split (good/rework) → Sink.

### Calculation Accuracy:
- ✅ Demand correctly propagates upstream through a 3-level assembly with BOM ratios (e.g., 4:1 material conversion).
- ✅ Split ratios correctly distribute required input when upstream demand is calculated.
- ✅ Utilization and bottleneck are correctly identified in complex networks.
- ✅ Test coverage includes multi-merge, multi-split, and complex cascading scenarios.

### User Adoption:
- ✅ Users can authenticate, save models, and own their data securely.
- ✅ Users can import a spreadsheet model and export results.
- ✅ Users receive clear error messages when attempting invalid graph structures (cycles, improper connections).
- ✅ Users can create, duplicate, and compare scenarios without data loss.

## 11. Implementation Priority & Risks

### Critical Path (Build First):
1. **DAG Graph Engine:** Support merges and splits with validation.
2. **Topological Sort & Complex Propagation:** Core calculation logic for real networks.
3. **Merge/Split Configuration UI:** BOM ratios and split percentage inputs.
4. **Authentication:** Secure user sessions and data ownership.
5. **Import/Export:** Unlock Excel migration and result sharing.

### Testing Priority:
- Heavy focus on **topological sort correctness** with edge cases.
- BOM and split ratio interactions (potential source of bugs).
- Real-world scenario validation with actual manufacturing examples.

### Known Risks:
- **Graph Validation Complexity:** Detecting invalid merges/splits is more complex than V1 linear constraints. Needs rigorous testing.
- **Propagation Edge Cases:** Multi-level merges with different yields create intricate math. Test thoroughly before shipping.
- **User Confusion:** Clear documentation on how BOM and split ratios work is essential; confusing UI can break trust.
