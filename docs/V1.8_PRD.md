## 1. Overview

Enable users to create, manage, and test multiple "what-if" scenarios with instant simulation feedback. Users can switch between scenarios, edit node parameters, and instantly see bottleneck changes on the canvas. Simple metrics display (throughput, bottleneck, utilization) helps users compare outcomes. All scenarios persist to database.

---

## 2. Problem Solved

### Current Pain Points

- **One scenario at a time:** Users can only model one process configuration; testing variations requires manual re-setup
- **No scenario memory:** Changes aren't saved; users lose work or can't return to previous setups
- **Slow iteration:** Can't quickly compare "what if we add 1 machine?" vs "what if we change efficiency?"
- **Unclear bottleneck impact:** Hard to see how parameter changes affect bottleneck location and process health

### User Need

*"I need to quickly test multiple 'what-if' scenarios and see which one works best, without losing my work."*

### Solution

Scenario Manager MVP with:
- Scenario tabs/switcher (Scenario 1, Scenario 2, +New)
- Edit mode: Click node → change params → auto-simulate
- Instant feedback: Canvas updates, bottleneck re-highlighted
- Simple metrics: Throughput, bottleneck node, per-node utilization
- DB persistence: Save/load scenarios

## 3. Core Features (ALL IMPLEMENTED IN V1.0)

### 3.1 Scenario Switching

**UI Location:** Top of editor, next to model name

```
Model: Widget Assembly Line

[Scenario 1] [Scenario 2] [Scenario 3] [+ New Scenario]
```

**Behavior:**
- Clicking a scenario tab loads that scenario's full state (all nodes, parameters, results)
- "[+ New Scenario]" button duplicates the current scenario
- Unlimited scenarios per model
- Visual indicator shows which scenario is active (bold, highlighted)

**Creating a New Scenario by Duplication:**
- Click "[+ New Scenario]" → Duplicates current scenario
- New scenario created with name: "New Scenario"
- Rename field immediately opens (inline, highlighted)
- User types new name (e.g., "+1 Assembly Worker", "High Demand Test")
- [Save] or [Enter] creates scenario with new name
- No "Copy" or numbered suffix in name

**Unsaved Changes:**
- If user has pending edits and clicks another scenario tab:
  - Show modal: "Save changes to [Scenario Name]?"
  - Options: [Save] [Discard] [Cancel]

### 3.3 Simulation Auto-Run

**Trigger:** User changes ANY parameter and clicks [Simulate]

**What Happens:**
1. Calculation runs (uses existing V1.6 simulation engine)
2. Canvas graph updates immediately:
   - Node throughputs recalculated
   - Bottleneck re-identified (highest utilization)
   - Bottleneck node highlighted in red (pulsing, from V1.7)
3. Metrics panel updates (see 3.5)
4. No DB save yet (user can discard or [Save Scenario])

**Performance:** Simulation should complete <300ms for typical models (100 nodes)

---

### 3.4 Bottleneck Highlighting

**Visual Indicator:**
- Bottleneck node rendered in red with pulsing border (existing V1.7 feature)
- Stops pulsing on hover/focus (no distraction during editing)

**Metrics Callout:**
Below the metrics panel:
```
Bottleneck: Assembly (100% utilization)
Throughput limited by: Assembly capacity
```

**Edge Cases:**
- **Multiple bottlenecks (tied):** "Assembly & QC (both 100%)" — highlight both, note in metrics
- **No bottleneck:** If all nodes <80% → "No bottleneck (process healthy)"
- **Cascade:** If changing one node's capacity shifts bottleneck to another → metrics show new bottleneck

### 3.6 Save/Load Scenarios to DB

**Save Flow:**

1. User modifies scenario (parameters, demand)
2. Clicks [Simulate] → graph updates
3. Clicks [Save Scenario] button (always visible, top of panel)
   - Modal: "Save changes to [Scenario Name]?"
   - Options: [Save] [Discard] [Cancel]
4. On confirm:
   - Scenario data written to DB (all nodes, parameters, demand, results)
   - Button grays out (no unsaved changes)
   - Toast: "Scenario saved"

**Load Flow:**

1. User clicks scenario tab (e.g., "Scenario 2")
2. System loads from DB:
   - Node configuration (machines, efficiency, hours/day, etc.)
   - Global demand
   - Derived results (throughput, bottleneck, utilization)
3. Canvas renders immediately
4. Metrics panel displays saved results

**Data Structure:**

```
Scenario {
  id: string
  model_id: string
  name: string
  created_at: timestamp
  updated_at: timestamp

  data: {
    globalDemand: number
    nodes: [
      {
        id: string
        type: "source" | "process" | "sink"
        parameters: { machines, efficiency, hoursPerDay, ... }
      }
    ]
  }

  results: {
    throughput: number
    bottleneck_node_id: string
    cycle_time: number
    utilization: { [node_id]: number }
  }
}
```

## 4. User Workflows

### Workflow 1: "Test if adding 1 machine helps"

1. Open Base Scenario
2. Click [+ New Scenario] → "Base + 1 Machine" (duplicate created)
3. Click Assembly node → Parameter panel opens
4. Change "Machines: 4 → 5"
5. Click [Simulate]
   - Graph updates
   - Bottleneck shifts (Assembly 100% → 95%)
   - Metrics show new throughput
6. [Save Scenario]
7. Click [Compare] to see "Base" vs "Base + 1 Machine"
8. Decide: "Yes, add machine" or test another scenario

### Workflow 2: "What if demand increases 20%?"

1. Click "Scenario 1" tab
2. Parameter panel shows Global Demand: 40
3. Change to 48
4. Click [Simulate]
5. Canvas updates, bottleneck may shift
6. Metrics show impact
7. [Save Scenario] or [Discard] and try different change

### Workflow 3: "Compare 2 options for stakeholder meeting"

1. Have 2-3 scenarios ready (tested, saved)
2. Click [Compare] → Select "Scenario A" vs "Scenario B"
3. View side-by-side
4. Export best option to CSV
5. Email metrics to team

## 5. Edge Cases & Error Handling

### Edge Case 1: User Deletes a Scenario

- Show confirmation: "Delete [Scenario Name]? This cannot be undone."
- After delete: If no scenarios remain, create "Untitled Scenario 1" (blank)

### Edge Case 2: Multiple Bottlenecks (Tied Utilization)

- Display: "Bottleneck: Assembly & QC (both 100%)"
- Highlight both nodes in red on canvas

### Edge Case 3: No Bottleneck (All Nodes Healthy)

- Bottleneck text: "None (process at <80% utilization)"
- No red highlighting

### Edge Case 4: User Edits Scenario, Closes Tab Without Saving

- Modal warning: "You have unsaved changes in [Scenario Name]. Save?"
- [Save] [Discard] [Cancel]

## 6. Technical Architecture

### 7.1 Scenario State Management

Scenarios stored in Zustand store + Supabase DB:

```
ScenarioStore {
  scenarios: Scenario[]
  current_scenario_id: string

  actions: {
    loadScenario(id)
    createScenario(name)
    updateScenario(id, data)
    deleteScenario(id)
    saveScenarioToDB()
    loadAllScenariosFromDB()
  }
}
```

### 7.2 Simulation Trigger

On parameter change:
1. User clicks [Simulate]
2. Call `SimulationEngine.run(scenario.data)`
3. Derive results: throughput, bottleneck, utilization
4. Update store (not DB yet)
5. Canvas re-renders
6. Metrics update

### 7.3 DB Schema

```sql
CREATE TABLE scenarios (
  id UUID PRIMARY KEY,
  model_id UUID REFERENCES models(id),
  name VARCHAR(255),
  created_at TIMESTAMP DEFAULT NOW(),
  updated_at TIMESTAMP DEFAULT NOW(),

  data JSONB,  -- node config, demand, parameters
  results JSONB  -- throughput, bottleneck, utilization
);

CREATE INDEX idx_scenarios_model_id ON scenarios(model_id);
```
