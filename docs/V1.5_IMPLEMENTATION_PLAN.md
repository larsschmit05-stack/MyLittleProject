# V1.5 Implementation Plan - Complex Network Capacity Modeler

This document defines the build sequence for V1.5. It is intended for **Claude Sonnet (Implementation Engineer)** to execute step by step, and for **Codex (Engineering Reviewer)** to review against `docs/V1.5_PRD.md`.

**Core goal:** Extend the V1 linear-chain model to support DAGs with merges (assembly) and loss-based splits (scrap/defect), enabling modeling of 80% of real production lines.

---

## Scope Decisions (Pre-Implementation)

Before building, the following decisions narrow scope and prevent ambiguous implementations.

### Split Definition: Real Output vs. Loss/Scrap Edges
V1.5 supports both **loss-based splits** (scrap/defects) and **distribution splits** (byproducts/multiple markets).

- A split node can have multiple output edges, each classified as either:
  - **Real output edge:** Carries downstream demand in propagation (e.g., "95% good product to next process")
  - **Loss/scrap edge:** Visual indicator only, carries no demand (e.g., "5% waste to disposal")
- Both types contribute to the split ratio (must sum to 100%), but only real edges drive upstream calculations.
- A node with multiple real output edges (e.g., "80% Customer A, 20% Secondary Market") propagates demand through each independently.
- Loss edges are excluded from topological sort; they are computed last as a byproduct of the node's throughput.

This allows modeling of byproducts, multiple markets, and waste simultaneously while keeping propagation unambiguous.

### Graph Topology: Single Sink, Fork-Join Allowed
- **Exactly one Sink node** is required. Multiple final outputs are not supported in V1.5.
- **Fork-join is allowed:** A Process node may have multiple inputs (merge) AND multiple outputs (split) simultaneously.
  - Example: Assembly operation (merges 3 parts) followed by testing (splits into 95% pass + 5% fail).
  - Demand propagates through all paths independently via topological sort.
- All nodes must be reachable from at least one Source and must have a path that reaches the Sink. Orphaned subgraphs are a validation error, not a warning.

### Material Identity: Optional Display Field Only
- An optional `outputMaterial?: string` field is added to Process and Source nodes for labeling purposes in the UI.
  - Example: Source outputs "Raw Steel", Process outputs "Cut Steel", Merge outputs "Assembled Unit".
  - This is for user reference and UI clarity only; no material compatibility validation is enforced.
- No material compatibility validation is performed in V1.5. That is deferred to a later release.
- BOM ratios must be defined for **all** incoming edges on a merge node, or the graph is invalid (no silent 1:1 defaults; users must be explicit).

## Step 1: Type System & Domain Model Updates [COMPLETED]

**Goal:** Extend the existing type definitions to support BOM ratios, scrap edge marking, and the optional material field — without breaking V1 types.

**What will be built:**

- Extend `ProcessNodeData` in `types/flow.ts`:
  - Add `bomRatios?: Record<string, number>` — keyed by incoming edge ID; value = units of this input required per unit of output. Only meaningful when the node has ≥2 incoming edges.
  - Add `outputMaterial?: string` — optional display label for what this node produces (e.g., "Assembled Unit").
  - The existing `yield` field already handles split/loss math. No `splitRatios` field is added.

- Extend `SourceNodeData`:
  - Add `outputMaterial?: string` — optional display label for the raw material this source provides (e.g., "Raw Steel").

- Extend edge data (React Flow `Edge`):
  - Add `isScrap?: boolean` — marks this edge as a visual scrap/loss path. Scrap edges are excluded from demand propagation and must target a dead-end node (no outgoing edges from the scrap target, and the target is not the Sink).

- No new node types are introduced. Merges and scrap paths are configuration states of existing node types.

- `SerializedModel` requires no structural change. Supabase's `data: jsonb` column absorbs the new fields. V1 models loaded without these fields must treat them as `undefined` (empty), which is safe.

**Success Criteria:**
- All existing V1 tests still pass.
- A `ProcessNodeData` object with `bomRatios` compiles without TypeScript errors.
- An edge object with `isScrap: true` compiles without TypeScript errors.
- V1 models loaded from Supabase without these new fields do not throw.

**Risks:**
- `bomRatios` is keyed by **edge ID**, not source node ID. Edge IDs are generated by React Flow at connection time and change if the edge is deleted and recreated. Stale edge keys become orphaned data. Mitigation: in `onEdgesChange`, when edges are removed, delete the corresponding keys from `bomRatios` on the affected target node.
- `isScrap` on edges is new edge data. React Flow's default edge rendering ignores unknown data. Ensure the edge type or data is respected when rendering and validating.

## Step 2: Graph Validation Rewrite

**Goal:** Replace the V1 linear-only validation with DAG-aware rules that allow merges and scrap paths while enforcing the explicit V1.5 graph constraints.

**What will be built:**

**Rewrite `isValidConnection()` in `lib/flow/validation.ts`:**
- Remove: one-input / one-output restriction per node.
- Keep: no self-loops, no connections to/from wrong node types, no duplicate edges.
- Keep: no cycles — DFS-based cycle detection on the **real edges only** (existing edges + proposed new edge) before committing. Scrap edges are excluded from cycle check.
- Keep: Source cannot have an incoming edge; Sink cannot have an outgoing edge.
- Allow: Fork-join nodes (multiple inputs + multiple outputs) — no restriction.
- Add: Scrap edges (marked `isScrap: true`) cannot connect to nodes with further outgoing edges. Scrap is a dead-end.

**Rewrite `validateGraph()` in `lib/flow/validation.ts`:**
- Remove: the "single continuous linear path" check.
- Keep: at least 1 Source node.
- Replace "at least 1 Sink" with **exactly 1 Sink**. Two Sink nodes is an error, not a warning.
- Add: every non-Source node must be reachable from a Source (using real edges only; no floating subgraphs).
- Add: every non-Sink node must have a path to the Sink (using real edges only; no dead-end real nodes).
- Add: scrap edge targets must have no outgoing edges (enforces scrap is dead-end constraint).
- Add: for each merge node (≥2 incoming real edges), `bomRatios` must contain a positive value for every incoming edge ID. Missing or zero BOM ratios are an error.
- Add: for each split node (≥2 outgoing edges), split ratios must sum to 100% ± 1% (tolerance for floating point). Missing ratios are an error.
- Allow: fork-join nodes (≥2 real inputs + ≥2 real outputs). No restriction.

**Update `ValidationResult`:** Add typed error categories: `cycle`, `missing_bom`, `invalid_ratio_sum`, `invalid_sink_count`, `orphaned_node`, `invalid_scrap_target`.

**Success Criteria:**
- A valid linear `Source → Process → Sink` chain passes.
- A valid merge `Source A → Process → Merge ← Process ← Source B → Sink` passes.
- A valid fork-join `Sources → Assembly → Testing (95% pass, 5% fail) → Sink` passes.
- A valid scrap path `Source → Process (yield 95%) → Sink` with a scrap edge on the process node passes.
- A graph with 2 Sinks is rejected with `invalid_sink_count`.
- A cycle is rejected with `cycle`.
- A merge node with missing BOM ratios is rejected with `missing_bom`.
- A split node with ratios summing to 97% is rejected with `invalid_ratio_sum`.
- A scrap edge whose target has an outgoing edge is rejected with `invalid_scrap_target`.

**Risks:**
- Cycle detection must run speculatively (before the edge is stored) because `onConnect` fires before state persists. Pass `[...existingEdges, proposedEdge]` into the DFS check.
- Existing V1 validation tests cover linear-only rules. Separate the tests that remain valid (cycle detection, Source/Sink type restrictions) from those that must be rewritten (one-in/one-out per node). Do not delete V1 tests; mark them as updated.
- The "max 1 real output" rule at connection time depends on distinguishing scrap vs. real edges. This requires `isScrap` to be set on the edge before or during the connection attempt. Define a clear moment in the UI where the user marks an edge as scrap (see Step 5).

## Step 3: Topological Sort & Complex Demand Propagation

**Goal:** Replace the V1 linear chain-walking algorithm with a general DAG propagation engine supporting merges, real output splits, and loss edges.

**What will be built:**

**New function `topologicalSort(nodes, edges): string[] | null`** in `utils/calculations.ts`:
- Implements Kahn's algorithm (in-degree BFS), using **only real edges** (`isScrap !== true`).
- Loss/scrap edges are ignored; they don't affect node ordering.
- Returns node IDs ordered from Source(s) to Sink.
- Returns `null` if a cycle is detected in real edges (acts as a second safety net after validation).

**New function `calculateFlowDAG(model: SerializedModel): FlowResult`** (replaces `calculateFlow` for V1.5):

1. Filter edges: separate real edges from loss/scrap edges (`isScrap !== true` vs `isScrap === true`).
2. Run `topologicalSort` on real edges only. Return EMPTY_RESULT if it returns `null`.
3. Set the Sink node's `requiredThroughput = globalDemand`.
4. Walk nodes in **reverse topological order** (Sink → Sources), using **only real edges**:
   - **At a single-input node:** propagate `requiredInput = requiredThroughput / (yield / 100)` to upstream.
   - **At a merge node** (≥2 incoming real edges): for each incoming edge, propagate `requiredInput × bomRatio[edgeId]` to the upstream node via that edge. Each upstream node receives its individual demand based on its BOM ratio.
   - **At a split node** (multiple real output edges): demand arrives from each downstream real edge independently. The node's required throughput is the maximum demand across all paths (bottleneck path determines total input needed). Then apply yield: `requiredInput = max(downstreamDemands) / (yield / 100)`.
   - **Loss edges:** Computed after the fact as a byproduct. A node with [95% real output, 5% loss] produces: `node.requiredThroughput × 0.95` as real output, `node.requiredThroughput × 0.05` as loss (waste, not further demand).
5. Calculate effective capacity for each node: `throughputRate × availableTime × numberOfResources × (yield / 100)`.
6. Utilization = `requiredThroughput / effectiveCapacity`. Bottleneck = node with highest utilization.
7. Return `FlowResult` with per-node results and bottleneck ID.

**Keep V1 `calculateFlow()` intact.** `calculateFlowDAG()` is used exclusively for V1.5. Update `useFlowStore.ts` to call `calculateFlowDAG()`.

**Key Math:**
- **Merge BOM example:** Node requires 100 output units. BOM = `{ edgeA: 4, edgeB: 1 }`. Upstream source A must supply 400 units; source B must supply 100 units.
- **Scrap loss split example:** Process node has `yield = 95` with split [95% real output, 5% scrap]. Sink demands 95 units. Required input = `95 / 0.95 = 100 units`. The scrap edge is visual; it carries no demand upstream, but 5 units are produced and wasted.
- **Multi-market split example:** Process with split [80% to Customer A (real), 20% to Secondary Market (real)]. Customer A demands 80 units, Secondary Market demands 20 units. Node must produce `max(80/0.80, 20/0.20) = 100 units`. If yield = 100%, required input = 100. If yield = 90%, required input = `100 / 0.90 ≈ 111`.
- **Fork-join example:** Assembly (3 inputs) → Testing (95% pass, 5% fail). Merge calculates required inputs via BOM. Testing receives 100 units from assembly, outputs 95 to next step (demand-driven) and 5 as scrap (byproduct).

**Success Criteria:**
- A 3-source assembly `3×Source → 3×Process → Merge(Process) → Sink` with BOM `4:1:2` and demand 100 produces:
  - Merge node: requires 100 output units
  - Process feeding Source A: requires 400 units
  - Process feeding Source B: requires 100 units
  - Process feeding Source C: requires 200 units
- A scrap-split `Source → Process (yield 95%, scrap edge) → Sink` with Sink demand 95 produces:
  - Process node: `requiredThroughput = 95`, `requiredInput = 100`
  - Scrap edge: 5 units produced (byproduct, not demand-driven)
- A fork-join `3×Source → Assembly → Testing (95% pass, 5% fail) → Sink` with Sink demand 95 produces:
  - Testing node: requires 100 input, outputs 95 pass (demand-driven), 5 fail (scrap)
  - Assembly merges 3 sources with BOM, outputs 100 to Testing
  - Correct propagation through both merge and split
- A multi-market split `Source → Process → [80% Customer A, 20% Secondary]` where both are real outputs, with Customer A demanding 80 units produces:
  - Process: requires 100 output (80 / 0.80 if no yield)
  - Source: provides 100 units
- V1 single-chain models produce **identical** results with the new engine.
- Scrap/loss edges do not appear in topological sort and do not receive demand.

**Risks:**
- **Multi-level cascading (merge → split → merge):** Demand propagates independently through each path. The reverse-topological walk must strictly respect node order. A single out-of-order process produces silently wrong numbers. Test multi-level cascades explicitly in Step 4.
- **Merge node accumulation:** When multiple upstream edges converge on a merge node, each edge contributes independently via BOM. The propagation assigns `bomRatio[edgeId] × nodeRequiredThroughput` to each upstream source separately. This must be clearly documented in code comments to prevent confusion.
- **Split with multiple real outputs:** A split node with two real output edges (not scrap) receives demand from BOTH downstream paths. The node must produce enough to satisfy both demands. The calculation is: `requiredInput = max(demand_path1 / ratio1, demand_path2 / ratio2) / yield`. This is subtle and must be tested thoroughly.
- **Loss edge byproduct calculation:** Loss edges are computed AFTER demand propagation, as a byproduct of the node's throughput. A node with [95% real, 5% loss] produces 5 units of loss for every 100 units total. This is visual/informational, not demand-driven. Code must clearly separate these concepts.
- **V1 model compatibility:** V1 models have no `isScrap` on edges and no `bomRatios` on nodes. The engine must handle `undefined` bomRatios by treating a single-input node as 1:1 (the V1 behavior). Add a guard: `if (!bomRatios || incomingEdges.length === 1) → use 1:1`.

## Step 4: Calculation Engine Tests

**Goal:** Prove the V1.5 engine against all defined scenarios before building UI on top of it.

**What will be built:**

Test suite additions to `utils/calculations.test.ts`:

- **`topologicalSort()` tests:**
  - Single linear chain → correct order
  - Diamond (two sources → merge → sink) → valid order (sources before merge)
  - Graph with cycle → returns `null`
  - Scrap edges are excluded from sort (scrap target does not appear in sorted output)

- **Merge / BOM tests:**
  - 3-source assembly with BOM `4:1:2` → each source receives correct required throughput
  - Merge node with `yield = 90%` → yield applies after BOM multiplication
  - 2-level merge cascade (merge A feeds into merge B) → propagation is correct at both levels

- **Split / scrap tests:**
  - Process node with `yield = 95%` and scrap edge → required input is `demand / 0.95`; scrap edge carries no demand
  - Process node with `yield = 100%` → no amplification (regression guard)
  - Chain with yield at every step → cumulative loss correctly propagates upstream

- **Multi-output real splits:**
  - Process with two real outputs [80% Customer A, 20% Secondary] where Customer A demands 80 and Secondary demands 20 → node produces 100 (max throughput satisfies both)
  - Process with [80% real, 20% scrap] where downstream demands 80 → node produces 100 (80/0.80 if no other yield)
  - Process with fork-join (merge input → split output [95% pass, 5% fail]) → demand correctly flows through assembly, then splits

- **Fork-join / cascading tests:**
  - Assembly (3 inputs, BOM 4:1:1) → Testing (95% pass, 5% fail) → Sink with 95 unit demand
    - Testing requires 100 input, assembly requires 400+100+100 = 600 from sources
  - Multi-level: Merge → Process → Split → Merge → Sink

- **Bottleneck tests:**
  - Bottleneck is in a merge branch (not the main chain) → correctly identified
  - Bottleneck is at a split node with multiple real outputs → correctly identified
  - All nodes at equal utilization → any node is returned as bottleneck (deterministic)

- **V1 regression tests:** All existing V1 tests must pass without modification.

**Success Criteria:**
- All new and existing tests pass.
- At least one test covers a 2-level merge cascade.
- At least one test covers a scrap edge being excluded from demand.
- Zero regression on V1 scenarios.

**Risks:**
- Floating-point accumulation in multi-level BOM math. Use `.toBeCloseTo(value, 4)` throughout (not exact equality).
- Test helper functions (`makeSource`, `makeProcess`, `makeSink`) need parameters for `bomRatios`, `outputMaterial`, and edge `isScrap`. Extend helpers in-place rather than duplicating them. Maintain backward compatibility for existing callers.

## Step 5: Merge Configuration UI & Scrap Edge UX

**Goal:** Let users configure BOM ratios on merge nodes and mark edges as scrap from the right sidebar and canvas.

**What will be built:**

**In `PropertiesPanel.tsx` — new "Bill of Materials" section (shown when node has ≥2 incoming real edges):**
- Header: "Bill of Materials"
- For each incoming edge: one labeled numeric input
  - Label: `"From: [upstream node name]"` (look up via edge source ID → node name)
  - Value: `bomRatios[edgeId]` or `1` as the default pre-filled value
  - On change: `updateNodeData(nodeId, { bomRatios: { ...existing, [edgeId]: newValue } })`
- Helper text: "Units of this input required per 1 unit of output"
- Validation inline: show a red error message if any ratio is 0 or negative

**In `PropertiesPanel.tsx` — new "Output Material" field (shown for all Process and Source nodes):**
- A single text input labeled "Output Material"
- Value: `outputMaterial` from node data or blank
- On change: `updateNodeData(nodeId, { outputMaterial: value })`
- Helper text: "What does this node produce? (e.g., 'Cut Steel', 'Assembled Unit')"
- This is a display/labeling field only — no validation against other nodes

**Scrap edge marking:**
- When a user selects an edge on the canvas, show an "Edge Properties" section in the right sidebar (currently no edge config exists).
- Section contains a single checkbox: "This is a scrap/loss path (no downstream demand)"
- On toggle: update the edge's `data.isScrap` boolean in the store
- Visual distinction: scrap edges render as dashed lines (using React Flow's `style` or a custom edge type)

**Single-input / single-output nodes show neither the BOM section nor any scrap toggle** — no regression for V1 users.

**Success Criteria:**
- Selecting a merge node (≥2 incoming real edges) shows the BOM section with labeled inputs.
- Changing a BOM ratio immediately triggers recalculation.
- Selecting any edge and marking it as scrap renders it dashed and excludes it from demand propagation.
- The "Output Material" field saves and displays correctly for Process and Source nodes.
- All V1 node configurations (no merge, no scrap) show no new UI sections.

**Risks:**
- Edge selection and edge properties in the right sidebar is new territory — `PropertiesPanel` currently only reacts to selected nodes. Zustand `selectedEdgeId` state must be added alongside `selectedNodeId`. These are mutually exclusive: selecting a node deselects any edge, and vice versa.
- The "max 1 real output" connection rule (Step 2) depends on `isScrap` being set on the edge. If a user draws an edge and does not mark it as scrap, the validation treats it as a real edge. Make the default behavior clearly "real" and let users opt-in to scrap. Do not auto-detect scrap based on topology.
- Node name lookups for BOM labels must be reactive. If the upstream node's name is changed, the BOM label must update without requiring a re-select of the merge node.

## Step 6: Error Feedback & Edge Case Polish

**Goal:** Surface all new validation errors clearly and ensure the results panel degrades gracefully on invalid graphs.

**What will be built:**

- Display validation errors from `validateGraph()` in the editor as a dismissible inline banner below the canvas toolbar. One message at a time (prioritize the most critical error):
  - `cycle`: "A cycle was detected. Remove the highlighted edge to continue."
  - `missing_bom`: "BOM ratios are missing on [Node Name]. Open the node to configure them."
  - `invalid_ratio_sum`: "Split ratios on [Node Name] sum to [X]% but must equal 100%."
  - `invalid_sink_count`: "The graph must have exactly one Sink node."
  - `orphaned_node`: "[Node Name] is not connected to the main flow."
  - `invalid_scrap_target`: "A scrap edge from [Node Name] connects to a node with further outputs. Scrap paths must be dead-ends."

- Results panel: when `calculateFlowDAG` returns EMPTY_RESULT, show `"Cannot calculate: [reason]"` using the error category from `validateGraph`. No blank fields, no NaN, no zeros that look like real results.

- Canvas node display for merge nodes: show a small "BOM" badge on the node when `bomRatios` is configured and valid. Show an orange indicator when BOM is incomplete (merge node exists but ratios are missing).

**Success Criteria:**
- Each error category surfaces a readable, actionable message referencing the node by name (not ID).
- Results panel never shows NaN or blank values — always shows either valid results or a clear error state.
- The BOM badge on merge nodes updates immediately when ratios are added or removed.
- Validation errors do not block canvas editing — they only block calculation.

**Risks:**
- Error messages must use node names. At render time, always resolve node IDs to names before displaying. If a name is blank, fall back to the node ID.
- The banner must not stack multiple error messages for the same error category. Deduplicate before rendering.

## Cross-Step Risks

- **Backward compatibility:** V1 models in Supabase have no `bomRatios`, `outputMaterial`, or `isScrap`. All load paths must treat `undefined` as the V1 default (1:1 BOM, no material label, no scrap edges). The engine guard in Step 3 handles this.
- **Edge ID stability:** `bomRatios` is keyed by edge ID. Deleting and reconnecting an edge generates a new ID, orphaning the old ratio. The purge in `onEdgesChange` (Step 1) is critical to prevent stale data accumulating silently.
- **Scrap edge marking as a manual step:** The system does not auto-classify edges as scrap based on topology. Users explicitly mark edges as scrap. This means a user could draw a "real" edge to a dead-end node and the system would treat it as a real output — causing a validation error rather than silently misclassifying. This is the correct behavior, but it requires clear UX guidance in Step 5.
- **Performance:** Topological sort and propagation are O(N+E). For realistic manufacturing models (< 50 nodes), this is not a concern. No optimization needed.
