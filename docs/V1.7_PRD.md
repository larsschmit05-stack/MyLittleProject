### 2.3 V1.7 Enhancement 2: Integrated Results Display

**Floating panel layout (top to bottom):**
1. Header: Node name + X button + Esc hint
2. Parameter controls (sliders, inputs, validators)
3. **Results section** (new):
   - Before/After comparison
   - System metrics
   - Selected node metrics

**Before/After Comparison Display:**

```
IMPACT SUMMARY
────────────────────────────────────────
Throughput:    150 units/hr  →  165 units/hr  ↑ +10% (green)
Bottleneck:    Packaging     →  Assembly      (shifted)
This Node:     78%           →  92%           ↑ +14% (green)
```

**Logic:**
- Displays current (saved) values vs. temporary (unsaved) parameter state
- Compute delta: `(after - before) / before * 100`
- Color: Green if metric improves, Red if worsens, Gray if no change
- Update immediately on parameter slider/input change (500ms debounce)

### 2.4 V1.7 Enhancement 3: Reset Button

**New button in floating panel: [Reset]**

**Behavior:**
- Reverts all unsaved changes back to last saved values
- Does NOT save to database
- Disabled when no unsaved changes (no dirty state)
- Clears before/after comparison display

**Workflow example:**
1. Sidebar shows Resources: 4 people (saved in DB)
2. User clicks "Float Panel" → Floating modal opens
3. User drags Resources slider: 4 → 6
4. Simulation auto-runs (500ms debounce, existing logic)
5. Before/After displays: "150 → 165 units/hr ↑"
6. User clicks [Reset] → Reverts to 4, before/after clears
7. Or clicks [Save Changes] → Commits to DB (existing button)
8. Or closes modal → Changes discarded (existing behavior)

### 2.5 No Changes to Simulation Engine

- Debouncing: 500ms (unchanged)
- Live simulation trigger: On debounce complete (unchanged)
- Auto-save logic: When user commits changes (unchanged)
- Parameter validation: Existing validation rules apply

### 3.2 V1.7 Enhancement 1: Pulse Animation (Visual Emphasis)

**Current:** Static red "BOTTLENECK" label + glow shadow
**Add:** Subtle scale pulse animation to draw eye to constraint

**Animation spec:**
- **Trigger:** Automatically on all bottleneck nodes after simulation
- **Motion:** Scale 1.0 → 1.05 → 1.0 (ease-in-out)
- **Duration:** 1.5 seconds per cycle
- **Stops when:** User hovers over node or node is selected
- **Intensity:** Very subtle (not distracting after 30+ mins of use)

**CSS Implementation:**
```css
.bottleneck-badge {
  animation: pulse 1.5s ease-in-out infinite;
}

.bottleneck-badge:hover,
.bottleneck-badge.selected {
  animation: none;  /* Stop when user focuses */
}

@keyframes pulse {
  0%, 100% { transform: scale(1); }
  50% { transform: scale(1.05); }
}
```

### 3.3 V1.7 Enhancement 2: Edge Case Handling & Messaging

**Current:** Only shows badge if bottleneckNodeId is set

**Add messaging & styling for edge cases:**

**Case 1: No Bottleneck** (all nodes balanced, no single constraint)
- **Detection:** If max utilization < 90% or all nodes within 5% utilization
- **Action:** Don't show red badge on any node
- **Message in Results section:** "✓ System is balanced — no bottleneck"
- **Optional:** Soft yellow "BALANCED" badge on top 1-2 nodes (nice-to-have)

**Case 2: Multiple Bottlenecks** (2+ nodes at ≥95% utilization)
- **Action:** Highlight ALL bottleneck nodes with red badge
- **Badge text:** Change to "BOTTLENECK (2/3)" to show count
- **Message in Results:** "Multiple bottlenecks: Assembly, QC both at 95%+ utilization"

**Case 3: Invalid Scenario** (0% yield, negative resources, broken validation)
- **Behavior:** Allow simulation to run (testing invalid states is OK)
- **Badge color:** Yellow instead of red
- **Badge text:** "⚠️ INVALID"
- **Message in Results:** "⚠️ This scenario contains invalid parameters. Results may not be realistic."

**Case 4: Empty or No-Process Graph**
- **Action:** No bottleneck analysis possible
- **Message in Results:** "Build a model to see bottleneck analysis"

## 7. Dependencies & Risks

### Prerequisites (Already Complete ✅)
- V1.6 complete: Simulation engine, model persistence (Supabase), RLS auth
- Parameter controls already exist: Resources, Yield, Labels
- Bottleneck detection algorithm already works

### Risks & Mitigations

| Risk | Impact | Mitigation |
|------|--------|-----------|
| Floating panel CSS positioning breaks on narrow desktop (< 1024px) | Panel overlaps graph too much | Test on 1024×768; adjust panel width if needed; default to mobile modal below 1024px |
| Pulse animation feels jarring or distracting | Users complain about visual noise | Keep scale subtle (1.0 → 1.05); test easing; make easily disableable in settings (future) |
| Reset button reverts changes but UI doesn't update immediately | User confusion: "Did it reset?" | Clear visual feedback on [Reset] click: flash badge, clear before/after, update UI instantly |
| Before/After shows stale metrics if simulation is still running | Confusing comparison display | Show loading state while simulation runs; disable before/after if simulation in-flight |
| Mobile bottom-sheet blocks canvas during interaction | User can't see graph while adjusting | Use swipe-down to peek at graph; optional: half-height sheet (if technical feasibility) |
| Multiple bottlenecks badge formatting confusing | User misinterprets "(2/3)" notation | Test labeling with users; consider tooltip: "2 of 3 nodes at bottleneck" |
| Edge case: Invalid scenario shows yellow badge but also simulation results | Confusing signal: is this valid or not? | Clear messaging in Results: "⚠️ Invalid parameters detected. Results shown for testing only." |

## 8. Use Cases & Value Proposition

### Use Case 1: Quick What-If Analysis with Visual Feedback

**Scenario:** Operations manager wants to test: "What if we add 2 people to Assembly?"

**V1.6 Workflow:**
1. Click Assembly node → Properties panel on right shows controls
2. Change "Resources" from 4 → 6
3. Simulation auto-runs (500ms debounce)
4. Look at "Results" section below to see new throughput and bottleneck
5. Compare mentally: "Old bottleneck was Packaging, now it's Assembly"

**V1.7 Enhancement (Auto-Float):**
1. Click Assembly node → Floating modal opens instantly over graph
2. Adjust slider: 4 → 6 (no extra button click needed)
3. Before/After shows side-by-side in panel: "150 → 165 units/hr ↑ +10%" and "Packaging → Assembly"
4. Red "BOTTLENECK" badge on Assembly with pulse animation draws eye
5. Instantly clear: Adding 2 people shifts constraint from Packaging to Assembly
6. Click [Reset] to try again without penalty, or [Save Changes] to commit
7. Press Esc to close, or click different node to switch

**Benefit:** No "Float Panel" button to find. One click (node) = edit it. Before/After removes cognitive load. Visual feedback (pulse) eliminates scanning. Keyboard close (Esc) enables fast iteration.

### Use Case 2: Mobile Scenario Testing (On-Site Decision Making)

**Scenario:** Plant manager on factory floor with tablet; needs to answer "Can we hit 200 units/hour?"

**V1.6 Limitation:**

- No easy way to undo bad parameter attempt

**V1.7 Enhancement (Auto-Float):**

4. Bottleneck badge visible with pulse animation
5. Swipe down to close, or press Esc, or tap X button
6. Can quickly test 3-4 scenarios without going back to desk

**Benefit:** Auto-open + mobile-friendly UX enables on-site decision making. No hunting for buttons. Reset button allows rapid "what-if without commitment" iteration.

### Use Case 3: Understanding Bottleneck Dynamics

**Scenario:** New process engineer learning: "Where's the bottleneck and what shifts it?"

**V1.6 State:**
- Red "BOTTLENECK" badge on Assembly (static)
- Engineer adjusts QC Yield: 92% → 95%

### Before/After Display Implementation

**Data sources (all in Zustand store):**
- Current parameters: `nodes[nodeId].data.numberOfResources`, `yield`, etc.
- Simulation results: `derivedResults.nodeResults[nodeId]`
- System metrics: `derivedResults.systemThroughput`, `bottleneckNodeId`

**Calculation:**
```typescript
// Saved (current) state
const beforeThroughput = derivedResults.systemThroughput;
const beforeBottleneck = derivedResults.bottleneckNodeId;

// Temporary (unsaved) state
// (same values - simulate would calculate new metrics on parameter change)

// Delta
const delta = ((afterThroughput - beforeThroughput) / beforeThroughput * 100).toFixed(1);
const color = delta > 0 ? 'green' : delta < 0 ? 'red' : 'gray';
```

### Reset Button Workflow

1. When floating panel opens: Store current parameter values as "saved state"
2. User adjusts slider/input: Update local state (not store yet)
3. Simulation auto-runs with temporary params (debounced)
4. If user clicks [Reset]: Revert all local state to saved state, update Zustand to match
5. If user clicks [Save Changes]: Commit temporary state to Zustand + database (existing logic)

### Performance Optimization

- Pulse animation: Pure CSS, 60fps (no JS overhead)
- Before/After: Computed instantly from store (no API call)
- Floating panel: Small component, renders in < 200ms
- Portal rendering: Standard React pattern, no performance issues
