# V1.9 PRD: Model Sharing & Permissions

**Status:** Ready for implementation
**Phase:** V1.9 Enhancement (Post V1.8 Scenario Manager)
**Date:** 2026-03-20

---

## 1. Overview

Enable users to share process models with team members via email invitations, granting **View** (read-only) or **Edit** (collaborative) access. Changes persist to the database and are visible to collaborators on refresh. This is async (non-real-time) collaboration.

---

## 2. Problem Solved

- Users work in isolation; cannot collaborate on a single model
- Team members must manually re-create or request models via email
- No visibility into who is using or editing which models
- Operations managers cannot delegate review of process designs

---

## 3. Core Features (ALL IN V1.0)

### 3.1 Share Button & Modal

**Entry Point:** Model detail page → Top toolbar → [Share] button

**Share Modal:**
```
Share Model: "Production Line v2"
─────────────────────────────
INVITE USERS:
  [Email input] [Role: View/Edit ▼] [Add User]

CURRENT ACCESS:
  bob@company.com (View) — Accepted
  charlie@company.com (Edit) — Pending (invited 3 days ago)
  [your_email@company.com] (Owner) — [remove disabled]

[Close]
```

### 3.2 Roles & Permissions

| Action | Owner | Edit | View |
|--------|-------|------|------|
| View model | ✓ | ✓ | ✓ |
| Edit nodes/parameters | ✓ | ✓ | ✗ |
| Create/edit scenarios | ✓ | ✓ | ✗ |
| Export model | ✓ | ✓ | ✓ |
| Delete model | ✓ | ✗ | ✗ |
| Manage sharing | ✓ | ✗ | ✗ |

**View Access:** Read-only UI — all input fields disabled, [Delete] and [Share] buttons hidden.

**Edit Access:** Full editing enabled except cannot delete model or manage sharing.

**Owner:** Full control. Only owner appears on access list (non-removable).

### 3.3 Invite & Email Flow

**Email Invitation (immediate):**
```
From: noreply@yourapp.com
To: alice@company.com
Subject: You've been invited to "Production Line v2" (Edit)

Hello Alice,

John has invited you to collaborate on "Production Line v2" with Edit access.

[Accept Invite →] (clickable link)

This invite expires in 30 days.
```

**Accept Flow:**
1. User clicks [Accept Invite] in email
2. If not logged in → signup/login form
3. If logged in → confirmation page with [Accept] [Decline]
4. Click [Accept] → redirected to model → can now edit

**Invite Status:**
- **Pending:** Waiting for user to accept (show "Invited X days ago")
- **Accepted:** User has full access (show "Accepted")
- **Expired:** After 30 days; owner can [Resend]
- **Declined:** User rejected; owner can re-invite

### 3.4 Concurrent Edits

**Last-Write-Wins:**
- User A edits Node X, User B edits Node Y → both save successfully
- User A and User B both edit Node X → last save persists, earlier change lost
- No conflict resolution; users must refresh to see others' changes
- **Acceptable for async collaboration**

### 3.5 Access Indicator

Non-owners see a badge on model detail page:
```
"Production Line v2"  👥 Shared with you (Edit)
```

Clicking badge opens read-only share modal (cannot manage users).

## 4. Database Schema

**New Table: model_access**

```sql
CREATE TABLE model_access (
  id UUID PRIMARY KEY,
  model_id UUID NOT NULL (FK: models.id),
  user_id UUID NOT NULL (FK: auth.users.id),
  role ENUM ('owner', 'edit', 'view') NOT NULL,
  status ENUM ('pending', 'accepted', 'declined', 'revoked') NOT NULL,
  invited_by UUID (FK: auth.users.id),
  invited_at TIMESTAMP,
  accepted_at TIMESTAMP,
  created_at TIMESTAMP,
  updated_at TIMESTAMP,

  UNIQUE (model_id, user_id),
  CONSTRAINT one_owner_per_model UNIQUE (model_id) WHERE role='owner'
);

CREATE INDEX idx_model_access_model ON model_access(model_id);
CREATE INDEX idx_model_access_user ON model_access(user_id);
```

**Row-Level Security:**
- Users can view model_access only for models they have access to
- Only owner can insert/update/delete model_access
- Non-owners see access status as read-only

## 5. API Endpoints

**POST /api/models/{modelId}/invite**
- Body: `{ email, role: 'view'|'edit' }`
- Sends email invitation immediately
- Returns: invitation details + status

**GET /api/models/{modelId}/access**
- Returns: list of users with access (owner only can see full details)
- Non-owners see limited view

**DELETE /api/models/{modelId}/access/{userId}**
- Revokes access (owner only)
- Updates status to 'revoked'

**POST /api/invites/{inviteToken}/accept**
- User accepts invite
- Sets status='accepted', accepted_at=now()
- Redirects to model detail page

## 7. Edge Cases & Error Handling

| Case | Behavior |
|------|----------|
| User tries to invite themselves | Error: "You cannot invite yourself" |
| User already has access | Error: "alice@company.com already has access" |
| Invited user doesn't exist yet | Send invite anyway; user signs up with that email |
| Invite expires (30 days) | Show [Resend] button in share modal |
| Owner deletes model | Access becomes invalid; other users see "Model no longer available" |
| User is revoked access | See "Your access was revoked"; redirected to dashboard |
| Two users edit same node simultaneously | Last write wins; no error shown (acceptable for async) |
