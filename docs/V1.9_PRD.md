# V1.9 PRD: Rework Loops

**Status:** Ready for Planning
**Phase:** V1.9 Enhancement (Post V1.8 Company Testing)
**Date:** 2026-03-18

---

## 1. Overview

Enable process modeling of **cyclic workflows** where output from a later process step returns to an earlier step for rework. Users can define rework percentages for any process node, and the system iteratively calculates steady-state throughput and resource utilization accounting for the overhead of rework cycles. Bottleneck detection and metrics are recalculated to show true impact of quality defects, yield losses, and material recycles.

**Key Addition:** Process models that currently only support linear and DAG flows will now support **simple rework loops** (single source of rework per node, converging cycles).

---

## 2. Problem Solved

### Current Pain Points

- **Ignores quality defects:** Models assume 100% yield; can't represent real-world defect rates (5-15% typical in manufacturing)
- **Inaccurate bottleneck detection:** Without accounting for rework, actual bottleneck may be masked (rework adds hidden load to earlier steps)
- **Unrealistic throughput predictions:** Scenarios don't account for extra processing time required to handle rework
- **No way to model yield impacts:** Can't test "What if we improve quality from 95% to 98%?" scenarios
- **Incomplete process simulation:** Many real processes have test-rework loops, assembly-rework flows, etc.

### User Need

*"I need to model processes with defects and rework cycles so I can see how quality impacts throughput and identify true bottlenecks."*

*"What if we reduce our defect rate from 5% to 2%? How much does that improve capacity?"*

### Solution

Rework Loops feature enabling:
- **Per-node rework percentage:** Define % of output that loops back to a specific earlier step
- **Iterative convergence simulation:** System automatically recalculates throughput and utilization accounting for rework until steady state
- **Updated bottleneck detection:** Identifies bottleneck after accounting for rework load
- **Rework impact metrics:** Shows total rework cycles, effective throughput vs. requested demand, utilization increase due to rework
- **Visual indicators:** Rework loop paths highlighted on canvas, rework percentages displayed on node connections

### 3.1 Define Rework Loops in Parameter Panel

**Entry Point:** Parameter panel for any PROCESS node

```
Parameter Panel (Right Sidebar)

┌─────────────────────────────────────┐
│ Editing: Assembly Node              │
├─────────────────────────────────────┤
│ Machines: [  5  ]                   │
│ Efficiency: [ 95 ]%                 │
│ Hours/Day: [ 16 ]                   │
│                                     │
│ REWORK SETTINGS                     │
│ ├─ Yield (% good output): [95]%     │
│ └─ Rework loops back to:            │
│    [Cutting v] [ 5 %] of output     │
│                                     │
│ [Simulate]                          │
└─────────────────────────────────────┘
```

**Rework Definition:**
- **Yield:** Percentage of output that is acceptable (100 - yield = defect rate)
  - Range: 0-100%
  - Default: 100% (no defects)
  - Example: Yield 95% = 5% rework rate

- **Rework Loop Target:** Which earlier node does the rework go to?
  - Dropdown: Lists all earlier nodes in process sequence
  - Can only select nodes that appear **before** current node in topological order
  - Default: None (no rework)

**Validation:**
- Can't create rework loops that form cycles in topology (except rework back cycles)
- Can't rework to a downstream node
- Can't rework to sink node
- Multiple nodes can rework to the same earlier node

### 3.3 Iterative Rework Calculation Engine

**Simulation Algorithm (New):**

When user clicks [Simulate]:

1. **Initialize:** Input demand at source = requested demand

2. **Iteration Loop (repeat until convergence):**
   - Run simulation as normal (V1.6 engine): traverse nodes, calculate throughput
   - Track rework loads: for each node with rework % > 0, calculate "rework quantity" = (output × rework%)
   - Feed rework quantities back to target nodes as additional input demand
   - Recalculate node throughputs with new demand
   - Check convergence: if all node throughputs changed <0.1% from last iteration, stop; else repeat

3. **Convergence:** Typically 3-5 iterations for typical models

4. **Final Results:**
   - Record final throughput (actual output at sink)
   - Record utilization (including rework load)
   - Record bottleneck
   - Store iteration count and convergence metrics

**Convergence Criteria:**
- Max iterations: 20 (safety limit to prevent infinite loops)
- Convergence threshold: All node throughputs within 0.1% of previous iteration
- Timeout: If convergence not reached in 20 iterations, alert user: "Rework loop did not converge. Check for unstable configuration."

**Example Calculation:**
```
Iteration 1:
- Source outputs 100 units/hr
- Cutting processes 100, yields 95 (5 rework) → Assembly
- Assembly processes 95, yields 95 (5 rework) → Cutting
- QC processes 95, yields 90 → Packaging
- Packaging: 90 units/hr (final throughput)

Iteration 2:
- Cutting now has [100 original + 5 rework = 105 input]
- Cutting processes 105, yields 95 → Assembly
- Assembly processes 95, yields 95 (5 rework) → Cutting
- QC processes 95, yields 90 → Packaging
- Packaging: 90 units/hr (final throughput)

Iteration 3: (no change, converged)
```

## 5. Edge Cases & Error Handling

### Edge Case 1: Convergence Failure

**Scenario:** Rework loop configuration is unstable (e.g., high rework %, causes exponential growth)

**Behavior:**
- Simulation runs max 20 iterations
- If not converged: Alert user with details
- Message: "⚠️ Rework loop did not converge after 20 iterations. Check: Are rework % values too high? Is target node downstream?"
- Show partial results (last good iteration)

### Edge Case 2: Multiple Rework Paths from One Node

**Scenario:** Assembly node has 5% rework to Cutting AND 3% rework to Wash

**Behavior:**
- Support allowed; system tracks both paths
- Convergence algorithm handles both simultaneously
- Metrics show both rework contributions
- Example: "Rework: 5% to Cutting, 3% to Wash (8% total from Assembly)"

### Edge Case 3: Cascading Rework (Chain Rework)

**Scenario:** Assembly → Cutting (5%), Cutting → Wash (3%)

**Behavior:**
- System handles chain rework correctly via iterative calculation
- Final rework metrics show combined effect
- Convergence typically needs more iterations but works correctly

### Edge Case 4: Rework Target is Deleted

**Scenario:** User deletes Cutting node while Assembly has rework set to Cutting

**Behavior:**
- Migration on delete: Clear rework settings for Assembly
- Show warning: "Assembly had rework to Cutting (now deleted). Rework settings cleared."
- No simulation error

### Edge Case 5: Rework % > 100%

**Scenario:** User tries to set Yield 80% and Rework 25% (total 105%)

**Behavior:**
- Validation error on [Simulate]
- Message: "Yield (80%) + Rework (25%) must equal ≤100%. Adjust yield or rework %."
- Block simulate until corrected

### Edge Case 6: No Rework Defined (All Yields = 100%)

**Behavior:**
- Convergence happens in 1 iteration (no rework feedback)
- Metrics show "Rework Cycles: 0"
- Normal simulation (backward compatible with V1.8)

### Edge Case 7: Rework Loop Back to Itself

**Scenario:** User tries to set Assembly to rework to itself (100% - 100%)

**Behavior:**
- Validation error: "Cannot rework to the same node"
- Block configuration

## 6. Technical Architecture

### 6.1 Data Model: Rework Configuration

```typescript
interface ReworkLoop {
  targetNodeId: string;  // Which earlier node does rework go to
  percentage: number;    // % of output that loops back (0-100)
}

interface ProcessNodeRework {
  yield: number;         // % of output that's good (0-100)
  reworkLoops: ReworkLoop[];  // Can have multiple rework targets
}

// Extended ProcessNode
interface ProcessNode {
  id: string;
  type: "process";
  // ... existing parameters (machines, efficiency, etc.)
  rework?: ProcessNodeRework;  // Optional; backward compatible
}
```

### 6.2 Rework Simulation Engine (New)

```typescript
interface ReworkSimulationResult {
  throughput: number;
  utilization: Map<string, number>;
  bottleneck: string;
  reworkCycles: number;        // Total units/hr in rework
  reworkRate: number;          // Rework as % of throughput
  convergenceIterations: number;
  converged: boolean;
}

function simulateWithRework(
  nodes: Node[],
  globalDemand: number,
  maxIterations: number = 20,
  convergenceThreshold: number = 0.001
): ReworkSimulationResult {

  let currentDemand = new Map<string, number>();
  currentDemand.set(sourceId, globalDemand);

  for (let iteration = 0; iteration < maxIterations; iteration++) {
    const previousThroughput = new Map(currentDemand);

    // Run normal simulation with current demand
    const result = runSimulation(nodes, currentDemand);

    // Calculate rework feedback
    const reworkFeedback = new Map<string, number>();
    for (const node of nodes) {
      if (node.type === "process" && node.rework?.reworkLoops.length > 0) {
        const nodeOutput = result.throughput.get(node.id) || 0;
        for (const rework of node.rework.reworkLoops) {
          const reworkAmount = nodeOutput * (rework.percentage / 100);
          reworkFeedback.set(
            rework.targetNodeId,
            (reworkFeedback.get(rework.targetNodeId) || 0) + reworkAmount
          );
        }
      }
    }

    // Update demand for next iteration
    for (const [nodeId, reworkAmount] of reworkFeedback) {
      currentDemand.set(nodeId, currentDemand.get(nodeId) || 0 + reworkAmount);
    }

    // Check convergence
    if (hasConverged(previousThroughput, currentDemand, convergenceThreshold)) {
      return {
        ...result,
        reworkCycles: calculateReworkCycles(reworkFeedback),
        convergenceIterations: iteration + 1,
        converged: true
      };
    }
  }

  // Max iterations reached
  return {
    ...lastResult,
    convergenceIterations: maxIterations,
    converged: false
  };
}
```
