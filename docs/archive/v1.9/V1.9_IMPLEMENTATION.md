## Database Schema & RLS (Days 1–2)

### Step 1: Create model_access Table Migration

**What to do:**
- Create a new Supabase migration file for the `model_access` table
- Define columns: id, model_id, user_id, role, status, invited_by, invited_at, accepted_at, created_at, updated_at
- Add indexes on (model_id, user_id) and (model_id) for query performance
- Add unique constraint on (model_id, user_id) to prevent duplicate access
- Add unique constraint on (model_id) WHERE role='owner' to enforce single owner

**File:** `supabase/migrations/[timestamp]_create_model_access.sql`

**Verification:**
- Migration runs without errors
- Table structure visible in Supabase dashboard
- Indexes created successfully

### Step 2: Create Invite Tokens Table

**What to do:**
- Create separate `invite_tokens` table to track invitation links
- Columns: token (unique), model_id, invited_email, role, status, expires_at, created_at
- This table holds the actual invite link data; model_access is created when accepted

**File:** `supabase/migrations/[timestamp]_create_invite_tokens.sql`

**Verification:**
- Table created
- Unique constraint on token working

### Step 3: Implement Row-Level Security (RLS) Policies

**What to do:**
- Enable RLS on both `model_access` and `invite_tokens` tables
- **model_access RLS:**
  - Users can VIEW model_access records only for models they have access to (JOIN with model_access WHERE user_id matches)
  - Users can INSERT model_access only if they're the owner of that model
  - Users can UPDATE model_access (e.g., change status) only if they're the owner
  - Users cannot DELETE model_access (no cascading deletes)
- **invite_tokens RLS:**
  - Anyone can VIEW invite_tokens by token (for accept flow)
  - Only owner can INSERT/UPDATE invite_tokens for their model
  - Only owner can revoke/delete

**File:** `supabase/migrations/[timestamp]_add_rls_model_access.sql`

**Verification:**
- RLS policies created and enabled
- Test that non-owners cannot view/modify other users' access records
- Test that anyone can view invite_tokens by exact token match

### Step 4: Add Helper Stored Procedures

**What to do:**
- Create PL/pgSQL function `get_user_access_level(model_id, user_id)` to quickly check if user has owner/edit/view access
- Returns role string or NULL if no access
- Create function `get_model_access_list(model_id)` to list all users with access (with RLS checks)

**File:** `supabase/migrations/[timestamp]_add_model_access_procedures.sql`

**Verification:**
- Functions execute without errors
- Return correct values for owner/edit/view cases

## Backend API Endpoints (Days 4–6)

### Step 10: POST /api/models/[modelId]/invite

**What to do:**
- Create endpoint to send an invite
- Request body: `{ email, role: 'view'|'edit' }`
- Validation:
  - Email format check
  - User is owner of this model
  - Email is not the owner's own email
  - User doesn't already have access
- Create invite_tokens record
- Generate unique invite token (use UUID or secure random)
- Set expiry to 30 days from now
- Call `sendInviteEmail()` to send email
- Return: `{ success, message, inviteToken }`

**File:** `pages/api/models/[modelId]/invite.ts`

**Verification:**
- Endpoint rejects non-owners (403)
- Endpoint rejects invalid emails (400)
- Endpoint rejects duplicate invites (400)
- Email is sent after successful create
- Response includes invite details

### Step 11: GET /api/models/[modelId]/access

**What to do:**
- Create endpoint to list all users with access to a model
- Check user has owner access (only owner can see full list)
- Return: `{ accessList: [ { email, role, status, invitedAt, acceptedAt } ] }`
- Order by invitedAt DESC
- Non-owners get empty list or 403

**File:** `pages/api/models/[modelId]/access.ts`

**Verification:**
- Owner sees complete list
- Non-owners see empty list or error
- Data is correctly sorted by date

### Step 12: DELETE /api/models/[modelId]/access/[userId]

**What to do:**
- Create endpoint to revoke access
- Check user is owner
- Update model_access record: status='revoked'
- Also invalidate any pending invites for this user
- Return: `{ success, message }`

**File:** `pages/api/models/[modelId]/access/[userId].ts`

**Verification:**
- Owner can revoke
- Non-owner gets 403
- Revoked user loses access immediately
- Revoked status persists in DB

### Step 13: POST /api/invites/[token]/accept

**What to do:**
- Create endpoint for user to accept an invite
- Get invite by token from invite_tokens table
- Check token hasn't expired (expires_at > now)
- Get or create user (if not logged in, may need to handle signup)
- Create model_access record: status='accepted', accepted_at=now
- Update invite_tokens: status='accepted'
- Return: `{ success, modelId, modelName }`
- Frontend will redirect to model detail page

**File:** `pages/api/invites/[token]/accept.ts`

**Verification:**
- Valid token returns success + model data
- Expired token returns error
- User account is created if needed
- model_access record is created
- Cannot accept same invite twice

### Step 14: POST /api/invites/[token]/decline

**What to do:**
- Create endpoint for user to decline an invite
- Get invite by token
- Update invite_tokens: status='declined'
- Do not create model_access record
- Return: `{ success, message }`

**File:** `pages/api/invites/[token]/decline.ts`

**Verification:**
- Declined invite updates in DB
- User is not added to model_access
- Same token can be re-invited later

### Step 15: POST /api/invites/[token]/resend

**What to do:**
- Create endpoint for owner to resend an expired invite
- Require user is owner (JWT check)
- Get invite by token
- Check invite is expired (expires_at < now)
- Generate new token
- Reset expires_at to 30 days from now
- Send email again
- Return: `{ success, newToken }`

**File:** `pages/api/invites/[token]/resend.ts`

**Verification:**
- Only owner can resend
- New token is unique
- Email is sent again
- Expiry is reset to 30 days out

## Risk Mitigation

| Risk | Mitigation |
|------|-----------|

| Invite token collisions | Use UUID v4 for tokens (collision probability ~0) |
| Concurrent edits losing data | Document last-write-wins behavior; consider V1.1 improvements |
| Performance degradation | Add indexes; profile database queries |
| Revoked access not immediate | Add optional polling (V1.1); log out user on refresh |
