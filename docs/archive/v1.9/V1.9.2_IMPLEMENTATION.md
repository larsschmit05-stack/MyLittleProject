# V1.9.2 Implementation Plan — Work Categories & Capacity Enhancements

## Overview

V1.9.2 delivers three focused features for multi-route, multi-category production modeling:

- **Full OEE inputs** — Availability, performance, quality as separate factors
- **Capacity-aware merge allocation** — Route demand to suppliers based on available capacity
- **Work categories** — Model 3–10 grouped demand streams (industry-agnostic: pharma, automotive, food, etc.)

This plan intentionally excludes push mode, dynamic split optimization, queueing, rework loops, and per-SKU multi-product complexity.

**Prerequisite:** V1.9.1 must be complete.

**Scope:** Industry-agnostic. Work categories and capacity-aware routing apply to any manufacturing/production process.

---

## Runs

| Run # | Name | Scope | Est. Time |
|-------|------|-------|-----------|
| 1 | Full OEE | OEE inputs + capacity formula update | 1 week |
| 2 | Capacity-Aware Allocation | Merge allocation by available capacity | 1–2 weeks |
| 3 | Work Categories | Grouped-demand layer for multiple demand streams | 1–2 weeks |

**Estimated total:** 3–5 weeks

Run 3 is optional and should only be implemented if users need to model multiple demand categories (pharma: batch sizes, automotive: model variants, food: product types, etc.).

---

## Run 1: Full OEE

**Goal:** Replace the single yield-style factor with separate availability, performance, and quality inputs.

### Changes

| File | Change |
|------|--------|
| `types/flow.ts` | Replace `yield` with `availabilityRate`, `performanceEfficiency`, `qualityRate` in process node data |
| `components/editor/PropertiesPanel.tsx` | Replace the single yield input with three OEE inputs |
| `utils/calculations.ts` | Update effective capacity formula to use all three factors |
| `components/editor/ResultsPanel.tsx` | Show compact OEE breakdown |
| `lib/flow/validation.ts` | Validate each OEE factor is 0-100 |
| `utils/calculations.test.ts` | Add tests for OEE formula and legacy migration |

### Deliverables

- separate OEE inputs in the UI
- updated effective capacity calculation
- no regression in utilization or bottleneck behavior

---

## Run 2: Capacity-Aware Allocation

**Goal:** Improve merge behavior when multiple upstream methods can supply the same downstream process state.

### Changes

| File | Change |
|------|--------|
| `types/flow.ts` | Add optional `capacityLimit` to relevant node data |
| `components/editor/PropertiesPanel.tsx` | Add `capacityLimit` input where applicable |
| `utils/calculations.ts` | Update merge allocation to use available capacity instead of only flow share |
| `lib/flow/validation.ts` | Validate positive capacity limits |
| `utils/calculations.test.ts` | Add merge-allocation and infeasibility tests |

### Deliverables

- demand can be routed across alternative upstream methods by available capacity
- infeasible upstream supply is detected clearly
- single-path and existing simple flows still behave as before

---

## Run 3: Work Categories (Industry-Agnostic Grouped Demand)

**Goal:** Add a light grouped-demand model for multiple work categories without introducing full per-SKU complexity.

### Scope Guard

This run models a small number of work categories (3–10), not hundreds or thousands of individual products. Examples:
- **Pharma:** Large batches, small batches, manual transfers
- **Automotive:** Model X, Model Y, variants
- **Food:** Recipe A, Recipe B, seasonal product

### Changes

| File | Change |
|------|--------|
| `types/flow.ts` | Add work category structure and grouped demand inputs |
| `store/useFlowStore.ts` | Add actions for creating and editing work categories |
| `components/editor/` | Add a compact UI for managing work-category demand |
| `utils/calculations.ts` | Propagate demand per category and aggregate utilization at shared nodes |
| `lib/flow/validation.ts` | Validate work-category demand and path consistency |
| `utils/calculations.test.ts` | Add work-category regression tests |

### Deliverables

- users can model multiple demand categories (pharma, automotive, food, any industry)
- shared resources correctly aggregate utilization across categories
- per-category bottleneck analysis shown in results
- backward compatibility for single-demand models is preserved

---

## Out of Scope

The following are intentionally excluded from this implementation plan:

- queueing and lead-time modeling
- rework loops
- supply-push mode
- dynamic split ratios
- full multi-product or per-SKU modeling

---

## Final Assessment

This implementation plan stays aligned with the current product direction: deterministic capacity modeling with better machine realism and better handling of alternative upstream routes. It is materially smaller, lower-risk, and better matched to the pharmacy process than the earlier V1.9.2 draft.
