# Product Requirements Document (V1.9.1) — Robustness & Clarity

## 1. Product Goal

V1.9.1 closes gaps in the V1.9 DAG calculation engine and UI messaging. The goal is to make infeasible scenarios explicit, eliminate threshold inconsistencies, remove legacy code, and clarify critical parameter semantics. **No new features; all improvements address detectability, consistency, and maintainability.**

### Priority 1: Fix throughputRate Semantic Ambiguity

**Problem:** Users can easily misconfigure the `throughputRate` field because its semantics are ambiguous.

**Current behavior (from calculations.ts:106):**
```
effectiveCapacity = throughputRate × availableTime × numberOfResources × (yield / 100)
```

**Industry standard:** "Throughput rate" in manufacturing universally means **output rate** (good units per time).

**User misinterpretation:**
- User thinks: "My machine makes 100 good parts/hour" → enters 100
- System calculates: `100 (rate) × 8 (hours) × 1 (machine) × 0.8 (80% yield) = 640 units`
- **Wrong!** Should be: `100 (output rate) × 8 (hours) × 1 (machine) = 800 units` (no yield multiplication)

The issue: If throughputRate is already the **output rate**, multiplying by yield double-counts the scrap loss.

**Solution:** Clarify the label and add a tooltip in PropertiesPanel:

```typescript
<div className="space-y-2">
  <label>
    Processing Rate (Input)
    <HelpIcon content="Units processed per hour (before yield loss). If 100 units/hr go into the process and 80% survive, effective output is 80 units/hr." />
  </label>
  <input
    type="number"
    placeholder="E.g., 100 units/hour"
  />
  <small className="text-xs text-gray-500">
    This is the rate at which raw material enters the process, before accounting for scrap.
    The system applies yield loss to calculate good output rate.
  </small>
</div>
```

**Also update CLAUDE.md** to define this clearly:
```markdown
- **throughputRate**: Input units per time unit (before scrap/yield).
  E.g., 100 units/hr means 100 units of material enter the process.
  If yield is 80%, the good output is 80 units/hr.
  The formula: effectiveCapacity = throughputRate × availableTime × resources × (yield/100)
  accounts for this: input_rate × time × resources × yield_fraction = good_output_capacity
```

**Test case:**
```typescript
test('throughputRate is input rate, not output rate', () => {
  const model = {
    nodes: [
      { id: 'source', type: 'source', data: {} },
      {
        id: 'process',
        type: 'process',
        data: {
          throughputRate: 100,        // 100 units/hr input
          availableTime: 8,           // 8 hours
          numberOfResources: 1,
          yield: 80,                  // 80% survive
          conversionRatio: 1.0,
          outputMaterial: 'Product',
        },
      },
      { id: 'sink', type: 'sink', data: {} },
    ],
    edges: [
      { id: 'e1', source: 'source', target: 'process', data: {} },
      { id: 'e2', source: 'process', target: 'sink', data: {} },
    ],
    globalDemand: 640,  // Request the effective capacity
  };
  const result = calculateFlowDAG(model);
  // Effective capacity: 100 × 8 × 1 × 0.8 = 640 units
  expect(result.nodeResults['process'].effectiveCapacity).toBe(640);
  expect(result.nodeResults['process'].utilization).toBe(1.0); // At capacity
});
```

**Acceptance Criteria:**
- ✅ Field is relabeled "Processing Rate (Input)" or similar
- ✅ Tooltip clarifies "before yield loss"
- ✅ CLAUDE.md defines the semantics with an example
- ✅ Test verifies the math matches expected behavior

### Priority 2: Add Time-Unit Validation

```

- Available Time: 8 hours (user means hours)
- Yield: 100%
- Resources: 1

Calculated capacity: 10 × 8 × 1 × 1.0 = 80 units
Actual capacity: 10 units/min × 60 min/hr × 8 hrs = 4,800 units

Wrong by 60x — but no error, just silently incorrect numbers.
```

### Priority 3: Explicit Infeasibility Handling

**Problem:** The engine produces `Infinity` utilization for zero-capacity or impossible scenarios (utils/calculations.ts:400), but the UI largely collapses that to "N/A" or blank rather than an explicit "demand cannot be met" state. Impossible models look ambiguous instead of clearly invalid.

**Impact:** Users cannot distinguish between:
- A scenario that is balanced (system throughput = demand)
- A scenario that is bottlenecked (system throughput < demand)
- A scenario that is *infeasible* (a required upstream node has zero capacity but nonzero demand)

**Definition of Infeasible:**
- Node has `requiredThroughput > 0` AND `effectiveCapacity === 0`
- This happens when:
  - `throughputRate <= 0`, OR
  - `availableTime <= 0`, OR
  - `numberOfResources <= 0`, OR
  - `yield <= 0` (scrap factor makes output impossible)

**Solution:**
1. Add an `infeasible` flag to the `FlowResult` type:
   ```typescript
   export interface FlowResult {
     systemThroughput: number;
     bottleneckNodeId: string | null;
     nodeResults: Record<string, NodeResult>;
     infeasible: boolean;           // NEW
     infeasibleNodeIds: string[];   // NEW
   }
   ```

2. In `calculateFlowDAG()`, after computing node results, check for infeasibility:
   ```typescript
   const infeasibleNodeIds: string[] = [];
   for (const [nodeId, result] of Object.entries(nodeResults)) {
     if (result.requiredThroughput > 0 && result.effectiveCapacity === 0) {
       infeasibleNodeIds.push(nodeId);
     }
   }
   const infeasible = infeasibleNodeIds.length > 0;
   return { systemThroughput, bottleneckNodeId, nodeResults, infeasible, infeasibleNodeIds };
   ```

3. Update `useFlowStore` to surface infeasibility:
   ```typescript
   // In the Results Panel or Canvas status:
   if (flowResult.infeasible) {
     return (
       <Alert variant="destructive">
         <AlertCircle className="h-4 w-4" />
         <AlertTitle>Infeasible Scenario</AlertTitle>
         <AlertDescription>
           {flowResult.infeasibleNodeIds.map(id => nodeNameMap[id]).join(', ')}
           {flowResult.infeasibleNodeIds.length === 1 ? 'has' : 'have'} zero capacity
           but nonzero demand. Check throughput rate, available time, yield, or resources.
         </AlertDescription>
       </Alert>
     );
   }
   ```

4. Canvas rendering: Highlight infeasible nodes with a distinct visual state (hatching or border color).

**Acceptance Criteria:**
- ✅ `calculateFlowDAG()` returns `infeasible: true` and identifies problem nodes
- ✅ Infeasible nodes are highlighted on the canvas with a clear visual indicator
- ✅ Results panel shows an error message identifying the infeasible nodes and the cause
- ✅ Test: A process node with `throughputRate = 0` and `requiredThroughput > 0` is correctly flagged

### Priority 4: Centralize Utilization Thresholds

**Problem:** The bottleneck threshold (0.95) is used inconsistently across the app:
- Classification (`lib/flow/bottleneck.ts:11`): `>= 0.95`
- Canvas coloring: `> 0.95` (different comparison)
- PDF export: varies

The same 95% utilization case can render differently depending on which surface renders it.

**Solution:**
1. Create a constants file `lib/flow/thresholds.ts`:
   ```typescript

    * Utilization threshold for bottleneck classification.
    * Nodes at or above this threshold are considered bottlenecks.
    *
    * Standard: 0.95 = 95% utilization
    * Reasoning: >95% utilization indicates the node is resource-constrained
    * and cannot meaningfully increase throughput.
    */
   export const BOTTLENECK_THRESHOLD = 0.95;

   /**
    * Spread threshold for "balanced" classification.
    * If max utilization < 0.90 OR all nodes within 5% spread,
    * the system is considered balanced (no single bottleneck).
    */
   export const BALANCED_THRESHOLD = 0.90;
   export const BALANCED_SPREAD = 0.05;

   /**
    * Elevated threshold for classification.
    * If max utilization is between BALANCED_THRESHOLD and BOTTLENECK_THRESHOLD,
    * and spread > BALANCED_SPREAD, the system is "elevated" (high but not constrained).
    */
   // Derived: BALANCED_THRESHOLD < max < BOTTLENECK_THRESHOLD && spread > BALANCED_SPREAD
   ```

2. Update all usages to import from this file:
   - `lib/flow/bottleneck.ts`: Use `BOTTLENECK_THRESHOLD`, `BALANCED_THRESHOLD`, `BALANCED_SPREAD`
   - Canvas coloring logic: Same constants
   - PDF export: Same constants
   - Metrics/charts: Same constants

3. Add a comment to each file explaining why the threshold matters:
   ```typescript
   import { BOTTLENECK_THRESHOLD } from '../../lib/flow/thresholds';

   // Nodes at or above BOTTLENECK_THRESHOLD (95%) are the system's constraint.
   // They cannot meaningfully increase throughput without additional resources.
   const isBottleneck = utilization >= BOTTLENECK_THRESHOLD;
   ```

**Acceptance Criteria:**
- ✅ All threshold comparisons use the centralized constants
- ✅ A node with utilization = 0.95 renders the same way in canvas, results panel, and PDF
- ✅ Tests verify that threshold changes propagate consistently

### Priority 6: ConversionRatio Documentation & Tests

**Problem:** `conversionRatio` semantics are partially documented but still fragile. The UI label says "input units per output unit" in PropertiesPanel.tsx, merge nodes hide it, and the behavior is easy to misread.

**Current State:**
- Label in UI: "input units per output unit"
- Used in demand propagation: `demandToUpstream = (requiredOutput / yield) * conversionRatio`
- No concrete examples in tests or help text

**Solution:**
1. **Add a constants file with clear definition** (`lib/flow/conversionRatio.ts`):
   ```typescript
   /**
    * CONVERSION RATIO SEMANTICS
    *
    * conversionRatio = (input units) / (output units)
    *
    * This tells the system how many units of input are needed to produce one unit of output.
    *
    * EXAMPLES:
    *
    * Example 1: Bottling Process
    * - Input: Bulk liquid in a tank
    * - Output: 1 liter bottle
    * - Ratio: 1.05 (5% loss in filling/spillage)
    * - Meaning: To get 1 bottle, need 1.05 liters of bulk liquid
    *
    * Example 2: Forging (with waste)
    * - Input: Raw steel stock
    * - Output: 1 machined part
    * - Raw weight: 5 kg, Final part: 1 kg
    * - Ratio: 5.0
    * - Meaning: 5 kg of raw material → 1 kg finished part (80% scrap removed by machining)
    *
    * Example 3: Assembly (multiple inputs via BOM)
    * - Input: Individual parts (managed via BOM ratios at merge node)
    * - Output: 1 assembled unit
    * - This node has conversionRatio = 1.0 (no loss; assembly is 100% efficient)
    *
    * FORMULA (in demand propagation):
    * demandToUpstream = (requiredDownstreamOutput / yieldFraction) × conversionRatio
    *
    * Where:
    * - requiredDownstreamOutput: units needed from this node to satisfy demand
    * - yieldFraction: yield / 100 (e.g., 95% yield → 0.95)
    * - conversionRatio: input units per output unit
    *
    * EXAMPLE CALCULATION:
    *
    * Process node parameters:
    * - Throughput rate: 100 units/hour
    * - Available time: 8 hours/day
    * - Yield: 95% (5% scrap)
    * - Number of resources: 1 machine
    * - Conversion ratio: 1.05
    *
    * Demand from downstream: 100 units/day
    *
    * Step 1: Account for yield
    *   demandWithYield = 100 / 0.95 = 105.26 units (need this much output before scrap)
    *
    * Step 2: Account for conversion (material loss)
    *   demandFromUpstream = 105.26 × 1.05 = 110.53 units
    *   Meaning: Upstream must supply 110.53 units to account for material loss
    *
    * Step 3: Calculate effective capacity of this node
    *   effectiveCapacity = 100 units/hour × 8 hours × 1 machine × 0.95 yield = 760 units
    *
    * Step 4: Calculate utilization at this node
    *   utilization = 105.26 / 760 = 0.139 (13.9% utilized)
    *   This node is NOT the bottleneck; it has plenty of capacity.
    */

   // Export for re-use in test examples

     bottling: 1.05,
     forging: 5.0,
     assembly: 1.0,
   };
   ```

## 3. Documented Limitations (Not In-Scope Defects)

The following are **inherent modeling limitations** of the product. They are not bugs but design choices that should be documented to set user expectations:

- **Deterministic model ignores queueing effects**: At high utilization (>85%), real queueing effects dominate. Actual lead time grows non-linearly as utilization approaches 100% (per Factory Physics VUT equation). The model has no concept of lead time, WIP, or variability. A node shown at 90% utilization may actually have severe delays in practice. Consider adding a documentation note: *"This model is deterministic and does not account for queueing delays at high utilization. At >85%, verify results with detailed simulations."*

- **Simplified OEE**: Only quality (yield) is captured. Availability (unplanned downtime) and Performance Efficiency (minor stops, speed losses) must be manually embedded in `availableTime` and `throughputRate`. This is a simplification that matches the MVP scope but should be documented.

- **Single globalDemand**: Can only model one product's demand. Multi-product mix analysis isn't possible in a single model. Users must create separate models per product.

- **Static split ratios**: Output routing proportions are user-configured constants, not demand-responsive. If demand shifts, the split ratios don't rebalance (this is correct for fixed routing, but users should understand it).

- **conversionRatio ignored at merge nodes**: Merge nodes use BOM ratios, not conversionRatio. If a node with conversionRatio later becomes a merge (multiple inputs), the conversionRatio is silently ignored. This could confuse users. Document: *"ConversionRatio applies only to single-input nodes. At merge nodes, use BOM ratios instead."*

**Action**: Add a "Limitations & Assumptions" section to the in-app Help or product documentation. Do not treat these as bugs or defects in V1.9.1.

## 4. What is NOT in V1.9.1 Scope

The following items are **out of scope** and should not be addressed in this release:

- **Capacity-aware merge allocation**: Currently distributes demand by flow share. Capacity-aware routing is a modeling feature, not a defect.
- **Supply-push logic**: Current model uses demand-pull (global demand at sink, propagated upstream). This matches V1.5 PRD intent.
- **Rework loops**: Explicitly excluded from DAG constraint by V1.5 PRD. Cycles are not allowed.
- **Threshold configurability**: Centralize the threshold, but do not make it user-configurable. This keeps the product simple.
- **Stochastic or queueing simulation**: Addressing the deterministic model limitation would require a major architectural change. This is a post-V2.0 consideration.

## 9. Notes & Rationale

### Why CLAUDE.md Accuracy Matters:
False documentation about data structures is a maintainability catastrophe. The next developer who reads CLAUDE.md will waste hours debugging a non-existent migration function or introduce bugs by "fixing" something that isn't broken. Correcting it immediately prevents future work from building on false assumptions.

### Why ThroughputRate Clarity Matters:
This is a **semantic bug in the UX**, not the math. The field is correctly used as input rate in the formula, but the label "Throughput Rate" suggests output rate (the industry standard). Users will naturally enter output rate and get wrong capacity numbers. A clarifying label prevents this configuration error.

### Why Time-Unit Consistency Matters:
Silent failures are worse than errors. If a user mixes units (throughput in units/minute, available time in hours), the system produces confidently wrong results with no warning. Adding a required time-unit selection makes it impossible to make this mistake.

### Why Infeasibility Matters:
Impossible scenarios should be **obvious** to users, not ambiguous. A node with zero throughput capacity but nonzero demand is a red flag that the model is misconfigured (e.g., forgot to enter a throughput rate). Surfacing this explicitly prevents hours of debugging.

### Why Threshold Centralization Matters:
Inconsistent thresholds break user trust. If a node shows as "balanced" in the canvas but "bottleneck" in the PDF, the user will question the tool's correctness. A single source of truth eliminates this ambiguity.

### Why Legacy Cleanup Matters:
Dead code increases review burden and creates confusion about which engine is actually used. Removing it clarifies the codebase for future maintainers.

### Why ConversionRatio Docs Matter:
This parameter is easy to misinterpret (is it input-per-output or output-per-input?). Real examples (bottling, forging) give users a concrete mental model instead of forcing them to guess or compute examples by hand.
