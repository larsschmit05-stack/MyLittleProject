## Run 2: Model Structure — Fields & Validation

**Goal:** Add structural support for time units and infeasibility detection. These changes modify the data model and validation logic but do not touch UI.

### Priorities Included

- **P2**: Add Time-Unit Validation
- **P3**: Explicit Infeasibility Handling

### Key Changes

| File | Change | Type |
|------|--------|------|
| `types/flow.ts` | Add `timeUnit: 'minute' \| 'hour' \| 'day' \| 'week'` to `SerializedModel` | Type addition |
| `types/flow.ts` | Add `infeasible: boolean`, `infeasibleNodeIds: string[]` to `FlowResult` | Type addition |
| `lib/flow/validation.ts` | Add `validateTimeUnits()` function | New validation |
| `utils/calculations.ts` | Update `calculateFlowDAG()` to detect and return infeasibility | Logic enhancement |
| `store/useFlowStore.ts` | Initialize `timeUnit` field in new models | Store update |
| `utils/calculations.test.ts` | Add test: time-unit validation rejects models without `timeUnit` | Test |
| `utils/calculations.test.ts` | Add test: node with `throughputRate=0` and `requiredThroughput>0` is flagged `infeasible` | Test |

### Deliverables

- ✅ `SerializedModel` has `timeUnit` field (default: 'hour')
- ✅ `FlowResult` includes `infeasible: boolean` and `infeasibleNodeIds: string[]`
- ✅ `calculateFlowDAG()` detects infeasibility and returns it
- ✅ Validation rejects models without `timeUnit`
- ✅ Tests verify both time-unit and infeasibility detection
- ✅ Store initializes `timeUnit` on new models
- ✅ Tests pass (unit + integration)

### Dependencies

- **Depends on Run 1**: No hard dependency, but benefits from P0 docs clarity
- **Blocks Run 3 UI**: Run 3 will consume the new `infeasible` flag to render alerts

## Notes for Implementation

1. **Run 1 can be done immediately** — no blockers, low risk
2. **Run 2 should follow Run 1** — benefits from cleaned documentation
3. **Run 3 depends on Run 2** — consumes the new fields and flags
4. **Parallel work possible**: While Run 2 is being reviewed, Run 3 can be drafted (UI mockups, tests written without running them)
5. **Testing is sequential**: Run 1 tests must pass before Run 2 starts; Run 2 tests must pass before Run 3 testing
6. **All PRs should target `dev`** (not `main`)
