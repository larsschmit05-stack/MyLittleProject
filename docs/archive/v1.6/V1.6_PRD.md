# V1.6 Product Requirements Document

## Executive Summary

**V1.6** introduces user authentication and multi-tenant data persistence, transforming Operational Process Modeler from a stateless prototype into a proper multi-user SaaS application. Users can create accounts, save their capacity models securely, and access them across sessions.

This phase enables the core multi-tenant architecture necessary for production deployment.

---

## Product Vision Alignment

Operational Process Modeler is a visual SaaS tool for deterministic capacity flow modeling in production networks. V1.6 completes the foundation layer:

- **V1.0-V1.5:** Process modeling engine (linear flows, DAG support, merges/splits, validation, scenarios)
- **V1.6:** User authentication & multi-tenant data storage (THIS PHASE)
- **V1.7+:** Advanced features (rework loops, scenario templates, team collaboration)

V1.6 solves the critical problem: **Without authentication, users cannot securely save and own their work.**

## Phase Scope

### Must-Have Features (V1.6)

#### 1. Supabase Auth Setup (Email/Password)
**Impact:** High | **Effort:** 3 days | **Category:** Collaboration

Implement email/password authentication as the foundation for user identity management.

**What it solves:**
- Users can create accounts securely
- Application knows "who" is using it
- Foundation for all subsequent multi-user features
- Authentication state persists across sessions

**Key requirements:**
- **Signup flow:** Email, password (with validation), password confirmation
  - Email validation: Standard format checks
  - Confirmation email: Required (user must verify email before accessing app)
  - Password strength: Minimum 8 characters, mixed case required
  - Duplicate email handling: Clear error message
  - Auto-login after email verification
  - Clear messaging: "Check your email to confirm your account"

- **Login flow:** Email and password authentication
  - Session persistence: User stays logged in after page refresh
  - Invalid credentials: Generic message ("Invalid email or password")
  - Optional password reset link (can defer to V1.7)

- **Logout:** Clear session and redirect to login page

- **Auth state:** Zustand store for global auth state (consistent with existing state management)
  - `useAuthStore` provides: `user`, `loading`, `signup()`, `login()`, `logout()`
  - Used throughout app for route protection and user identification

- **Protected routes:** All application routes (dashboard, editor) require authentication
  - Next.js `middleware.ts` for server-side redirect of unauthenticated users to `/login`
  - Landing page `/` and `/login` remain public

**Technical implementation:**
- Supabase Email/Password provider
- `@supabase/supabase-js` (already installed)
- Next.js App Router middleware for route guards
- Zustand auth store (consistent with existing `useFlowStore` pattern)
- Components: `LoginPage`, `SignupPage`

**Success criteria:**
- Sign up works; user created in Supabase
- Login works; session persists on refresh
- Logout clears session and redirects
- Protected routes block unauthenticated access
- User ID available throughout app (for model ownership)
- Error messages are clear and helpful

#### 2. User Model Persistence (Supabase RLS)
**Impact:** High | **Effort:** 2 days | **Category:** Collaboration

Add `user_id` to the existing models table with row-level security, ensuring each user only accesses their own models.

**What it solves:**
- Multi-user safety: RLS prevents unauthorized access
- Users see only their own models on the dashboard
- Foundation for team sharing (V1.7+)

**Builds on existing code:**
- `lib/persistence.ts` — already has `insertModel`, `updateModel`, `fetchModel`, `listModels`, `renameModel`, `deleteModel`, `duplicateModel`
- `store/useFlowStore.ts` — already has `saveAsNewModel`, `updateSavedModel`, `loadModel` with auto-save and error handling
- `components/editor/EditorHeader.tsx` — already has save UI with status indicators

**Key requirements:**

- **Database migration:** Add `user_id` column to existing `models` table
  ```sql
  ALTER TABLE models ADD COLUMN user_id UUID REFERENCES auth.users(id) ON DELETE CASCADE;
  CREATE INDEX idx_models_user_id ON models(user_id);
  ALTER TABLE models ENABLE ROW LEVEL SECURITY;
  ```

- **Row-Level Security policies:**
  - SELECT: Users see only their own models
  - INSERT: Users can only create models with their own user_id
  - UPDATE: Users can only edit their own models
  - DELETE: Users can only delete their own models

- **Extend existing persistence layer:**
  - `insertModel` — add `user_id` from auth session
  - `listModels` — RLS auto-filters; no code change needed
  - Other CRUD functions — RLS auto-filters by user_id

- **Dashboard models table:**
  - **Columns:** Name (sortable, clickable), Created (date), Last Updated (date), Actions (Edit, Delete)
  - Load model: Click name to open in editor
  - New model: "Create New Model" button
  - Empty state: "No models yet. Create one to get started."

- **Error handling:**
  - RLS violations caught and reported clearly
  - Missing models: "Model not found or deleted"
  - Network errors: Queue for retry (defer offline support to V1.7)

**Success criteria:**
- Model saved with correct user_id
- Dashboard loads only current user's models
- RLS prevents cross-user access (verified by testing as different user)
- Save/update/delete work from existing UI
- Auto-save continues working with user_id
- Error messages guide user action

#### 3. Validation UI Enhancement

Add visual error feedback (modal + node highlighting) on top of the existing validation engine.

**What already exists (no rebuild needed):**

**What V1.6 adds (UI only):**

- Red border/warning icon on problem nodes in the canvas
- "Cannot simulate" state when validation fails (block simulation button)

**Success criteria:**
- Problem nodes visually highlighted on canvas
- Error modal shows count + actionable error list
- Simulation blocked when validation fails
