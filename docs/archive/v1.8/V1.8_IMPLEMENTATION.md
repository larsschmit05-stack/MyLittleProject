# V1.8 Implementation Plan: Scenario Manager

**Phase:** V1.8 Enhancement
**Objective:** Enable users to create, test, and compare multiple "what-if" scenarios with instant feedback
**Timeline Estimate:** 3-4 weeks

---

## Architecture Overview

```
Scenario Manager (V1.8)
├── UI Layer
│   ├── Scenario Tabs (top of editor)
│   ├── Parameter Panel (right sidebar)
│   ├── Metrics Display (below canvas)
│   ├── Rename Dialog (inline, modal)
│   └── Comparison View (overlay)
│
├── State Management (Zustand)
│   ├── Current scenario ID
│   ├── All scenarios (in-memory cache)
│   └── Unsaved edits tracking
│
├── Data Layer (Supabase)
│   ├── scenarios table (JSONB node config + results)
│   └── Queries (CRUD operations)
│
└── Logic Layer
    ├── Simulation engine (reuse V1.6)
    ├── Bottleneck detection
    └── PDF generation
```

### PHASE 1: Foundation & Data Model (Days 1-3)

#### Step 1.1: Extend Existing Store & Define Scenario Types
**What:** Extend `useFlowStore` to support scenario management and define scenario types
**Why:** Single store prevents two sources of truth; reuse existing patterns
**Achieves:** Type safety, consistent state management

**Files to Create/Modify:**
- `types/scenario.ts` (new)
  ```
  - Scenario interface: { id, name, model_id, data, results, created_at, updated_at }
  - ScenarioData wraps full model state: { nodes, edges, globalDemand }
  - ScenarioResults: { throughput, bottleneck_node_ids[], utilization map, cycle_time }
  ```

- `store/useFlowStore.ts` (modify)
  ```
  - Add state: scenarios[], currentScenarioId, unsavedBaseline
  - Add actions:
    - createScenario(name)
    - loadScenario(id)
    - updateScenario(id, data)
    - deleteScenario(id)
    - saveScenarioToDB()
    - duplicateScenario()
  - Add selectors: getCurrentScenario(), hasUnsavedChanges(), getUnsavedBaseline()
  ```

**Success Criteria:**
- ✅ TypeScript compiles without errors
- ✅ Scenario state integrated into existing store
- ✅ Bottleneck_node_ids array supports tied bottlenecks
- ✅ Saved baseline tracks state for dirty checking and reset

---

#### Step 1.2: Database Schema & Supabase Integration
**What:** Create scenarios table and CRUD queries
**Why:** Scenarios must persist across refreshes and testing sessions
**Achieves:** Reliable data storage

**Files to Create/Modify:**
- `lib/db/scenarios.ts` (new)
  ```
  - supabase queries:
    - getScenarios(modelId) → fetch all
    - getScenario(id) → fetch one
    - createScenario(data) → insert, return id
    - updateScenario(id, data) → update (upsert on conflict)
    - deleteScenario(id) → hard delete
  ```

- Database migration (if not already in place):
  ```sql
  CREATE TABLE scenarios (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    model_id UUID NOT NULL REFERENCES models(id) ON DELETE CASCADE,
    name VARCHAR(255) NOT NULL,
    created_at TIMESTAMP DEFAULT NOW(),
    updated_at TIMESTAMP DEFAULT NOW(),

    data JSONB NOT NULL,  -- { nodes, edges, globalDemand }
    results JSONB         -- { throughput, bottleneck_node_ids[], utilization, cycle_time }
  );

  CREATE INDEX idx_scenarios_model_id ON scenarios(model_id);
  ```

**Success Criteria:**
- ✅ Scenarios can be created, read, updated, deleted
- ✅ Full model state (nodes, edges, demand) stored in data JSONB
- ✅ Results include bottleneck_node_ids array for ties
- ✅ Queries handle missing/null results gracefully

**Why:** End user needs to know what limits capacity (critical insight)
**Achieves:** Clear process visibility

**Files to Create/Modify:**
- `lib/simulation/bottleneck.ts` (new)
  ```
  - Function: detectBottleneck(utilization: {[nodeId]: number}) → string[]
  - Logic: Return all nodeIds with max utilization (≥95%)
  - Edge cases:
    - Multiple bottlenecks (tied): Return all nodeIds in array
    - No bottleneck (all <80%): Return empty array
    - All nodes: Process entire utilization map
  ```

- `store/useFlowStore.ts` (modify)
  ```
  - After simulation, call detectBottleneck()
  - Store bottleneck_node_ids[] in results (array to support ties)
  - Available in metrics display
  ```

**Success Criteria:**
- ✅ Bottleneck correctly identified (highest utilization)
- ✅ Tied bottlenecks handled (both highlighted)
- ✅ No bottleneck state handled (shown as "Healthy")
