# Product Roadmap Document (V1.9.2) — Work Categories & Capacity Enhancements

**Status:** Approved (minimal scope, industry-agnostic)
**Target Release:** Post-V1.9.1
**Effort Estimate:** 3–5 weeks

---

## 1. Overview

V1.9.2 is a focused enhancement to support **multi-route production** with **capacity-aware allocation** and **explicit OEE decomposition**.

### Problem Solved

Manufacturing processes often have:
- **Multiple upstream methods** that produce the same output (e.g., auto processing, semi-auto, manual)
- **Multiple product categories** flowing through shared downstream resources
- **Equipment losses** (downtime, speed, quality) that should be modeled separately for bottleneck clarity

Current V1 cannot handle these scenarios efficiently.

### What V1.9.2 Delivers

1. **Full OEE inputs** — Separate availability, performance, quality for realistic machine modeling
2. **Capacity-aware merge allocation** — Route demand to upstream methods based on available capacity, not static split ratios
3. **Work categories** — Model multiple demand streams (3–10 categories) without per-SKU complexity

This version keeps the product focused on deterministic capacity modeling. It does **not** expand into optimization, stochastic simulation, dynamic routing, or cyclic process behavior.

---

## 2. Scope

### Feature 1: Full OEE Inputs

**Why it stays:** The described process uses machines where uptime, speed losses, and quality losses should be modeled separately.

**What it adds:**
- `availabilityRate`
- `performanceEfficiency`
- `qualityRate`

**Value:**
- better capacity realism for Rowadose and inspection machines
- clearer bottleneck diagnosis
- no change to the deterministic modeling approach

**Rough effort:** 1 week

---

### Feature 2: Capacity-Aware Merge Allocation

**Why it stays:** Multiple preparation methods can feed the same downstream state. Demand should be able to flow to the method with available capacity instead of being split only by static flow share.

**Example fit:**
- fully automatic strip processing
- semi-automatic strip processing
- manual pot-to-pot transfer

All three can produce the same "ready for production" output state.

**What it adds:**
- optional `capacityLimit` on relevant nodes
- merge allocation based on available capacity
- clear infeasibility detection when total upstream capacity is insufficient

**Value:**
- better representation of alternative preparation routes
- fewer manual scenario iterations
- more realistic bottleneck behavior upstream of production

**Rough effort:** 1-2 weeks

---

### Feature 3: Work Categories (Industry-Agnostic Grouped Demand)

**Why it stays in limited form:** Users often need to model multiple product types or demand streams flowing through the same production network. Modeling each as a separate SKU is unnecessary; grouping into 3–10 work categories is sufficient.

**Scope limit:**
- model a small number of work categories (3–10, not 1000+)
- do **not** build a full SKU-by-SKU multi-product optimizer

**Example work categories (applies to any industry):**

*Pharma:*
- Large strip batches (1000+ units/week)
- Small strip batches (100–999 units/week)
- Pot-to-pot transfers (<100 units/week)

*Automotive:*
- Model X (500 units/week)
- Model Y (300 units/week)
- Variants & custom orders (100 units/week)

*Food Manufacturing:*
- Recipe A (200 batches/week)
- Recipe B (150 batches/week)
- Seasonal product (50 batches/week)

**What it adds:**
- grouped demand inputs (per work category)
- per-category demand propagation through the same DAG
- aggregated utilization across categories at shared resources
- per-category bottleneck analysis

**Value:**
- supports meaningful demand segmentation across any industry
- avoids per-SKU complexity while still modeling realistic product mixes
- shows which work category is the system bottleneck

**Rough effort:** 1–2 weeks

---

## 3. Explicitly Out of Scope

The following ideas from the previous V1.9.2 draft are removed from this version:

- queueing and lead-time modeling
- rework loops and cyclic solving
- supply-push mode
- dynamic split ratios
- full per-SKU multi-product modeling

These features are not required for the target pharmacy process and would move the product beyond its current deterministic capacity-planning focus.

---

## 4. Recommended Implementation Order

1. Full OEE
2. Capacity-aware merge allocation
3. Optional product families, only if real user scenarios require grouped demand

---

## 5. Success Criteria

V1.9.2 is successful if users can:

- model alternative upstream preparation methods feeding the same downstream process
- represent machine losses more realistically with OEE factors
- identify bottlenecks across preparation, production, and inspection
- optionally group demand into a small number of meaningful families without modeling every medicine separately

---

## 6. Final Assessment

V1.9.2 should remain a focused extension of the current product, not a broad advanced-modeling release. The retained scope supports the pharmacy packaging workflow while preserving simplicity, maintainability, and the deterministic DAG-based architecture.
